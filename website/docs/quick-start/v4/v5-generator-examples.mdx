# 生成模式使用方式（推荐）

生成模式必须先区分 Tailwind CSS 主版本。`tailwindcss@4` 是 CSS-first 引擎，`weapp-tailwindcss@5` 默认接管它的样式生成，并直接输出小程序目标 CSS；`tailwindcss@3` 仍是 PostCSS / `tailwind.config.js` 链路，默认保持旧项目的后处理兼容方式，避免升级 `weapp-tailwindcss` 时被迫改造 Tailwind 配置。

核心变化是：Tailwind CSS v4 项目不需要再注册 `@tailwindcss/vite`、`@tailwindcss/postcss` 或其他 Tailwind 官方构建插件；Tailwind CSS v3 项目继续保留原来的 `tailwindcss` PostCSS 插件和 `content` 扫描配置，只让 `weapp-tailwindcss` 负责小程序转译。

:::tip 新项目优先选择生成模式
如果你正在接入 Tailwind CSS 或准备升级到 v5，优先从生成模式开始。它会直接生成小程序目标 CSS，并用 `classSet` 精确约束 JS / 模板转译范围。
:::

## 安装

Tailwind CSS v4 直出生成路径只需要安装 `tailwindcss` 和 `weapp-tailwindcss`，不需要安装 Tailwind 官方构建插件，也不需要再配置 `postinstall` 脚本。

选择 Tailwind CSS v4：

```bash npm2yarn
npm install -D tailwindcss@4 weapp-tailwindcss
```

选择 Tailwind CSS v3：

```bash npm2yarn
npm install -D tailwindcss@3 weapp-tailwindcss
```

Tailwind CSS v4 项目使用 CSS-first 入口和 `@source`，默认由 v5 生成模式直接输出目标端样式；Tailwind CSS v3 项目继续使用 `tailwind.config.js` 的 `content` 和 `theme`，并保持旧链路兼容。

## 版本行为差异

| Tailwind 主版本 | 默认行为 | 扫描配置 | Tailwind 官方插件 | 迁移建议 |
| --- | --- | --- | --- | --- |
| `tailwindcss@4` | `generator.mode = 'auto'` 默认直出小程序 CSS | CSS 入口里的 `@source` | 移除 `@tailwindcss/vite` / `@tailwindcss/postcss` | 推荐用于新项目或准备升级 Tailwind v4 的项目 |
| `tailwindcss@3` | `auto` 下保持旧 CSS 后处理链路 | `tailwind.config.js` 的 `content` | 保留 `tailwindcss` PostCSS 插件 | 旧项目只升级 `weapp-tailwindcss` 时优先选择 |
| `tailwindcss@3` + `force` | 强制尝试 v3 generator | `tailwind.config.js` / `@config` | 可按试验链路移除 | 只建议在单独分支验证，CI 通过后再合并 |

## 按框架选择入口

| 场景 | 文档 | 推荐入口 | 注册位置 | CSS 入口 |
| --- | --- | --- | --- | --- |
| uni-app Vue Vite | [uni-app 生成模式](./generator/uni-app) | `WeappTailwindcss` | `vite.config.ts` | `src/main.css` |
| uni-app Vue2 Webpack | [uni-app 生成模式](./generator/uni-app#uni-app-vue2-webpack) | `WeappTailwindcss` | `vue.config.js` 的 `configureWebpack` | `src/app.css` |
| Taro Vite | [Taro 生成模式](./generator/taro) | `WeappTailwindcss` | `config/index.ts` 的 `compiler.vitePlugins` | `src/app.css` |
| Taro Webpack | [Taro 生成模式](./generator/taro#taro-webpack) | `WeappTailwindcss` | `config/index.ts` 的 `mini.webpackChain` | `src/app.css` |
| weapp-vite | [weapp-vite 生成模式](./generator/weapp-vite) | `WeappTailwindcss` | `vite.config.ts` | `app.css` 或项目全局样式 |
| Mpx | [Mpx 生成模式](./generator/mpx) | PostCSS 默认导出 + Webpack `WeappTailwindcss` | `mpx.config.js` | `src/app.css` |

:::tip 配置建议
Tailwind CSS v4 生成路径下不要再把 `@tailwindcss/vite` 或 `@tailwindcss/postcss` 注册到实际项目里，否则会形成两套 Tailwind 生成链路。Tailwind CSS v3 旧项目不要照搬这一条，需要保留原有 `tailwindcss` PostCSS 插件。Vite、Webpack、Webpack4 与 Gulp 入口推荐导入大写 `WeappTailwindcss`；小写 `weappTailwindcss` 也可用，适合偏函数式命名的项目。PostCSS 入口继续使用默认导出。
:::

## CSS 入口

Tailwind CSS v4 项目的 CSS 入口建议使用 CSS-first 写法：

```css title="app.css"
@import "tailwindcss";
@source "./src/**/*.{vue,js,ts,wxml,tsx,jsx}";
@source not "./dist";
```

Tailwind CSS v3 项目继续把扫描范围写在 `tailwind.config.js`，并保留原有 PostCSS 中的 `tailwindcss` 插件：

```js title="tailwind.config.js"
export default {
  content: ['./src/**/*.{vue,js,ts,wxml,tsx,jsx}'],
  theme: {
    extend: {},
  },
}
```

```js title="postcss.config.js"
export default {
  plugins: {
    tailwindcss: {},
  },
}
```

默认小程序产物不需要写 `generator`。如果希望显式写出目标，可以只传 `target`：

```js
generator: {
  target: 'weapp',
}
```

不传 `generator` 时会使用默认的 `mode: 'auto'` 和 `target: 'weapp'`。对 Tailwind CSS v4 来说这会接管样式生成；对 Tailwind CSS v3 来说，`auto` 不会主动替换旧链路。小程序选择器兼容配置也有默认值，`cssChildCombinatorReplaceValue` 默认就是 `['view', 'text']`，只有项目需要改成自定义组件标签时才需要传 `styleOptions`。

CI 严格校验 Tailwind CSS v4 生成路径时可以额外传 `mode: 'force'`。这样生成失败会直接让构建失败，避免静默回退到旧 CSS 后处理链路。Tailwind CSS v3 项目只有在明确要试验 v3 generator 时才使用 `force`。

## Web 目标调试

同一入口也可以设置 `generator: { mode: 'force', target: 'web' }` 生成浏览器 CSS，用于多端调试或对比 Tailwind 原始产物。`target: 'web'` 不做小程序选择器转义，也不会移除浏览器需要的 hover media；小程序生产构建应使用 `target: 'weapp'`。

## 验证矩阵

建议在合并 v5 示例时至少跑：

```bash
pnpm --filter weapp-tailwindcss test
pnpm --filter @weapp-tailwindcss-demo/mpx-tailwindcss-v5 build
pnpm build:apps
pnpm build:demo
```

其中 Taro 与 uni-app 的聚合构建可能被 guard 跳过；需要严格验证时，在交互式终端显式设置 `TARO_BUILD_STRICT=1` 或 `UNI_BUILD_STRICT=1`。

## 部署后验证

如果文档站使用预览域名或 Cloudflare Git 部署，建议在部署完成后直接检查新路由和旧入口是否符合预期：

```bash
curl -I https://next.tw.icebreaker.top/docs/quick-start/v4/v5-generator-examples
curl -I https://next.tw.icebreaker.top/docs/quick-start/v4/generator/uni-app
curl -I https://next.tw.icebreaker.top/docs/quick-start/v4/generator/taro
```

这些路由应返回 `200`；如果线上还没有出现新页面，优先检查 Cloudflare 的构建分支、构建命令、部署命令和输出目录是否与仓库配置一致。
