# 生成模式使用方式（推荐）

生成模式必须先区分 Tailwind CSS 主版本。`tailwindcss@4` 是 CSS-first 引擎，扫描范围写在 CSS 入口的 `@source`；`tailwindcss@3` 仍使用 `tailwind.config.js` 的 `content`。两个版本都会默认由 `weapp-tailwindcss@5` 接管 Tailwind 样式生成，并直接输出小程序目标 CSS。

核心变化是：大部分 Vite / Webpack 项目只需要注册 `weapp-tailwindcss` 自己的构建器插件，也就是 `weapp-tailwindcss/vite` 或 `weapp-tailwindcss/webpack` 导出的 `WeappTailwindcss`。实际项目中不需要再注册 `@tailwindcss/vite`、`@tailwindcss/postcss`、`tailwindcss` PostCSS 插件或其他 Tailwind 官方构建插件；`weapp-tailwindcss` 会按当前 Tailwind 主版本读取配置并接管样式生成。

:::tip 新项目优先选择生成模式
如果你正在接入 Tailwind CSS 或准备升级到 v5，优先从生成模式开始。它会直接生成小程序目标 CSS，并用 `classSet` 精确约束 JS / 模板转译范围。
:::

## 安装

生成模式只需要安装 `tailwindcss` 和 `weapp-tailwindcss`，不需要安装 Tailwind 官方构建插件，也不需要再配置 `postinstall` 脚本。

选择 Tailwind CSS v4：

```bash npm2yarn
npm install -D tailwindcss@4 weapp-tailwindcss
```

选择 Tailwind CSS v3：

```bash npm2yarn
npm install -D tailwindcss@3 weapp-tailwindcss
```

Tailwind CSS v4 项目使用 CSS-first 入口和 `@source`，Tailwind CSS v3 项目继续使用 `tailwind.config.js` 的 `content` 和 `theme`；两者都会默认由 v5 生成模式直接输出目标端样式。

## 默认集成原则

优先把 Tailwind CSS 样式生成、CSS 小程序适配、模板类名转译和 JS 类名转译都交给同一个 `WeappTailwindcss` 构建器插件：

```ts title="Vite"
import { WeappTailwindcss } from 'weapp-tailwindcss/vite'

export default {
  plugins: [
    WeappTailwindcss(),
  ],
}
```

```js title="Webpack"
const { WeappTailwindcss } = require('weapp-tailwindcss/webpack')

module.exports = {
  configureWebpack(config) {
    config.plugins.push(new WeappTailwindcss())
  },
}
```

不要在同一个小程序构建里再额外注册官方 Tailwind 构建插件。也就是说：

- Vite 项目不再添加 `@tailwindcss/vite`。
- PostCSS 配置里不再添加 `@tailwindcss/postcss`。
- Tailwind CSS v3 项目不再添加 `tailwindcss` PostCSS 插件。
- `package.json` 不再添加 `postinstall: "weapp-tw patch"`。

`postcss.config.js` 可以继续保留项目自己的非 Tailwind 插件，例如 `postcss-import`、框架内置 PostCSS 插件或业务自定义插件；只需要移除 Tailwind 官方生成插件。`weapp-tw patch` 仍可用于旧 CSS 后处理链路或手动调试补丁状态，但生成模式项目不需要把它写进安装生命周期。

## 版本行为差异

| Tailwind 主版本 | 默认行为 | 扫描配置 | Tailwind 官方插件 | 迁移建议 |
| --- | --- | --- | --- | --- |
| `tailwindcss@4` | `generator.mode = 'auto'` 默认直出小程序 CSS | CSS 入口里的 `@source` | 移除 `@tailwindcss/vite` / `@tailwindcss/postcss` | 推荐用于新项目或准备升级 Tailwind v4 的项目 |
| `tailwindcss@3` | `generator.mode = 'auto'` 默认直出小程序 CSS | `tailwind.config.js` 的 `content` | 移除 `tailwindcss` PostCSS 插件 | 继续沿用 v3 配置心智 |

## 按框架选择入口

| 场景 | 文档 | 推荐入口 | 注册位置 | CSS 入口 |
| --- | --- | --- | --- | --- |
| uni-app Vue Vite | [uni-app 生成模式](./generator/uni-app) | `WeappTailwindcss` | `vite.config.ts` | `src/main.css` |
| uni-app Vue2 Webpack | [uni-app 生成模式](./generator/uni-app#uni-app-vue2-webpack) | `WeappTailwindcss` | `vue.config.js` 的 `configureWebpack` | `src/app.css` |
| Taro Vite | [Taro 生成模式](./generator/taro) | `WeappTailwindcss` | `config/index.ts` 的 `compiler.vitePlugins` | `src/app.css` |
| Taro Webpack | [Taro 生成模式](./generator/taro#taro-webpack) | `WeappTailwindcss` | `config/index.ts` 的 `mini.webpackChain` | `src/app.css` |
| weapp-vite | [weapp-vite 生成模式](./generator/weapp-vite) | `WeappTailwindcss` | `vite.config.ts` | `app.css` 或项目全局样式 |
| Mpx | [Mpx 生成模式](./generator/mpx) | `WeappTailwindcss` | `mpx.config.js` | `src/app.css` |

:::tip 配置建议
生成模式下不要再把 `@tailwindcss/vite`、`@tailwindcss/postcss` 或 `tailwindcss` PostCSS 插件注册到实际项目里，否则会形成两套 Tailwind 生成链路。Vite、Webpack、Webpack4 与 Gulp 入口推荐导入大写 `WeappTailwindcss`；小写 `weappTailwindcss` 也可用，适合偏函数式命名的项目。
:::

## CSS 入口

Tailwind CSS v4 项目的 CSS 入口建议使用 CSS-first 写法：

```css title="app.css"
@import "tailwindcss";
@source "./src/**/*.{vue,js,ts,wxml,tsx,jsx}";
@source not "./dist";
```

Tailwind CSS v3 项目继续把扫描范围写在 `tailwind.config.js`，CSS 入口继续使用 v3 的 `@tailwind` 指令：

```css title="app.css"
@tailwind base;
@tailwind components;
@tailwind utilities;
```

```js title="tailwind.config.js"
export default {
  content: ['./src/**/*.{vue,js,ts,wxml,tsx,jsx}'],
  theme: {
    extend: {},
  },
}
```

默认小程序产物不需要写 `generator`。Tailwind CSS v3 和 v4 都会接管样式生成；小程序选择器兼容配置也有默认值，`cssChildCombinatorReplaceValue` 默认就是 `['view', 'text']`，只有项目需要改成自定义组件标签时才需要传 `styleOptions`。生成失败会直接让构建失败，避免继续产出缺失样式的包。

## Web 目标调试

同一入口也可以设置 `generator: { target: 'web' }` 生成浏览器 CSS，用于多端调试或对比 Tailwind 原始产物。`target: 'web'` 不做小程序选择器转义，也不会移除浏览器需要的 hover media；小程序生产构建不需要显式配置 `target`。

## 验证矩阵

建议在合并 v5 示例时至少跑：

```bash
pnpm --filter weapp-tailwindcss test
pnpm --filter @weapp-tailwindcss-demo/mpx-tailwindcss-v5 build
pnpm build:apps
pnpm build:demo
```

其中 Taro 与 uni-app 的聚合构建可能被 guard 跳过；需要严格验证时，在交互式终端显式设置 `TARO_BUILD_STRICT=1` 或 `UNI_BUILD_STRICT=1`。

## 部署后验证

如果文档站使用预览域名或 Cloudflare Git 部署，建议在部署完成后直接检查新路由和旧入口是否符合预期：

```bash
curl -I https://next.tw.icebreaker.top/docs/quick-start/v4/v5-generator-examples
curl -I https://next.tw.icebreaker.top/docs/quick-start/v4/generator/uni-app
curl -I https://next.tw.icebreaker.top/docs/quick-start/v4/generator/taro
```

这些路由应返回 `200`；如果线上还没有出现新页面，优先检查 Cloudflare 的构建分支、构建命令、部署命令和输出目录是否与仓库配置一致。
