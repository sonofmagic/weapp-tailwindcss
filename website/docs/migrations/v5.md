# weapp-tailwindcss v4 到 v5 迁移指南

`weapp-tailwindcss@5` 的核心变化是：生成模式变成默认链路。升级后，小程序构建不再依赖 `weapp-tw patch` 写入 Tailwind 补丁，也不再建议把 Tailwind 官方 Vite / PostCSS 生成插件和 `WeappTailwindcss` 同时注册到同一个小程序构建里。

迁移时先判断项目使用的是 `tailwindcss@3` 还是 `tailwindcss@4`。两者都会由 `WeappTailwindcss` 接管样式生成，但配置入口不同：

| Tailwind 版本 | 样式入口 | 扫描配置 | v5 推荐链路 |
| --- | --- | --- | --- |
| Tailwind CSS 3.x | `@tailwind base; @tailwind components; @tailwind utilities;` | `tailwind.config.js` 的 `content` | 构建器插件读取 `cssEntries` 后生成小程序 CSS |
| Tailwind CSS 4.x | `@import "tailwindcss";` | CSS 入口中的 `@source` | 构建器插件读取 `cssEntries` 后生成小程序 CSS |

> 说明：Tailwind CSS v4 官方升级指南要求 Node.js 20+，并且 v4 使用更现代的浏览器目标。小程序项目如果需要兼容更老的 WebView，仍可以继续使用 `tailwindcss@3`，只把 `weapp-tailwindcss` 升级到 v5。

## 1. 升级前确认

升级前先整理当前项目属于哪一种形态：

| 当前形态 | 需要处理的重点 |
| --- | --- |
| `weapp-tailwindcss@4` + `tailwindcss@3` | 删除 patch 和 Tailwind PostCSS 生成插件，保留 `tailwind.config.js content` |
| `weapp-tailwindcss@4` + `tailwindcss@4` | 删除官方 Tailwind Vite / PostCSS 生成插件，改用 CSS-first 入口与 `cssEntries` |
| 已经有多个 Tailwind 入口 | 每个入口都放进 `cssEntries`，并用 `@config` 或独立配置隔离上下文 |
| H5 和小程序共用仓库 | H5 构建可继续用官方 Tailwind 插件，小程序构建只注册 `WeappTailwindcss` |

建议先在新分支升级，并保存一份可回滚的构建产物：

```bash
git switch -c migrate/weapp-tailwindcss-v5
pnpm install
pnpm build
```

## 2. 升级依赖

如果项目准备继续使用 Tailwind CSS 3.x：

```bash
pnpm add -D weapp-tailwindcss@5 tailwindcss@3
```

如果项目已经使用 Tailwind CSS 4.x，或准备同时迁移 Tailwind 主版本：

```bash
pnpm add -D weapp-tailwindcss@5 tailwindcss@4
```

Tailwind CSS v4 的官方 Vite / PostCSS 插件在 Web 构建中仍然有价值，但小程序构建中不再和 `WeappTailwindcss` 同时注册。也就是说，下面这些依赖只有在 H5 / Web 构建还直接使用官方 Tailwind 插件时才需要保留：

| 依赖 | 小程序构建是否需要 | 说明 |
| --- | --- | --- |
| `@tailwindcss/vite` | 不需要 | 小程序构建由 `weapp-tailwindcss/vite` 生成 CSS |
| `@tailwindcss/postcss` | 不需要 | 小程序构建不要再走 Tailwind 官方 PostCSS 生成链路 |
| `postcss-import` | 通常不需要 | Tailwind CSS v4 官方链路已内置 import 处理；业务 PostCSS 另有需求时再保留 |
| `autoprefixer` | 通常不需要 | Tailwind CSS v4 官方链路已内置前缀处理；业务另有浏览器目标时再保留 |

## 3. 删除旧 patch 链路

v5 生成模式不需要安装后 patch。迁移时删除 `package.json` 里的旧脚本：

```jsonc title="package.json"
{
  "scripts": {
    // 删除这一行
    "postinstall": "weapp-tw patch"
  }
}
```

如果项目里还依赖 `tailwindcss-patch`，也应删除：

```bash
pnpm remove tailwindcss-patch
```

`weapp-tw patch` 在 v5 中只保留兼容提示，不再负责写补丁记录、刷新缓存或隔离子包上下文。迁移后看到类似 patch 刷新日志，通常说明项目里还有旧脚本、旧插件或旧模板没有删干净。

## 4. 改造构建器插件

v5 的推荐配置是只注册 `WeappTailwindcss`，并通过 `cssEntries` 告诉插件哪些 CSS 入口需要参与 Tailwind 生成。

### 4.1 Vite / weapp-vite / uni-app Vite

```ts title="vite.config.ts"
import path from 'node:path'
import { defineConfig } from 'vite'
import { WeappTailwindcss } from 'weapp-tailwindcss/vite'

export default defineConfig({
  plugins: [
    WeappTailwindcss({
      rem2rpx: true,
      cssEntries: [
        path.resolve(__dirname, 'src/app.css'),
      ],
    }),
  ],
})
```

如果这是 `uni-app Vite` 项目，`WeappTailwindcss` 仍然放在 `uni()` 后面：

```ts title="vite.config.ts"
import path from 'node:path'
import uni from '@dcloudio/vite-plugin-uni'
import { defineConfig } from 'vite'
import { WeappTailwindcss } from 'weapp-tailwindcss/vite'

export default defineConfig({
  plugins: [
    uni(),
    WeappTailwindcss({
      rem2rpx: true,
      cssEntries: [
        path.resolve(__dirname, 'src/app.css'),
      ],
    }),
  ],
})
```

### 4.2 Taro Vite

```ts title="config/index.ts"
import path from 'node:path'
import { defineConfig } from '@tarojs/cli'
import { WeappTailwindcss } from 'weapp-tailwindcss/vite'

export default defineConfig({
  compiler: {
    type: 'vite',
    vitePlugins: [
      WeappTailwindcss({
        rem2rpx: true,
        appType: 'taro',
        cssEntries: [
          path.resolve(__dirname, '../src/app.css'),
        ],
      }),
    ],
  },
})
```

### 4.3 Webpack / Taro Webpack / Mpx

```js title="webpack.config.js"
const path = require('node:path')
const { WeappTailwindcss } = require('weapp-tailwindcss/webpack')

module.exports = {
  plugins: [
    new WeappTailwindcss({
      rem2rpx: true,
      cssEntries: [
        path.resolve(__dirname, 'src/app.css'),
      ],
    }),
  ],
}
```

Taro Webpack 项目通常写在 `mini.webpackChain`：

```js title="config/index.js"
const path = require('node:path')
const { WeappTailwindcss } = require('weapp-tailwindcss/webpack')

module.exports = {
  mini: {
    webpackChain(chain) {
      chain.merge({
        plugin: {
          install: {
            plugin: WeappTailwindcss,
            args: [{
              rem2rpx: true,
              cssEntries: [
                path.resolve(__dirname, '../src/app.css'),
              ],
            }],
          },
        },
      })
    },
  },
}
```

## 5. Tailwind CSS 3.x 项目迁移

Tailwind CSS 3.x 项目迁移到 v5 时，不需要改成 CSS-first。保留 `tailwind.config.js`，但移除 `postcss.config.js` 中的 Tailwind 生成插件。

```css title="src/app.css"
@tailwind base;
@tailwind components;
@tailwind utilities;
```

```js title="tailwind.config.js"
/** @type {import('tailwindcss').Config} */
module.exports = {
  content: [
    './src/**/*.{html,js,ts,jsx,tsx,vue,wxml,axml,ttml,mpx}',
    '!./src/uni_modules/**/*',
    '!./node_modules/**/*',
    '!./dist/**/*',
    '!./unpackage/**/*',
  ],
  theme: {
    extend: {},
  },
}
```

旧写法：

```js title="postcss.config.js"
module.exports = {
  plugins: {
    tailwindcss: {},
    autoprefixer: {},
  },
}
```

v5 推荐写法：

```js title="postcss.config.js"
module.exports = {
  plugins: {
    // 只保留业务自己的非 Tailwind PostCSS 插件
  },
}
```

## 6. Tailwind CSS 4.x 项目迁移

Tailwind CSS 4.x 项目迁移到 v5 时，入口 CSS 使用官方 CSS-first 写法，并让 `cssEntries` 指向这个纯 `.css` 文件。

```css title="src/app.css"
@import "tailwindcss";
@source "./**/*.{vue,js,ts,jsx,tsx,wxml,axml,ttml,mpx}";
@source not "../dist";
@source not "../unpackage";
@source not "../node_modules";
```

如果仍需要复用旧的 `tailwind.config.js`，使用 `@config` 显式加载：

```css title="src/app.css"
@config "../tailwind.config.js";
@import "tailwindcss";
@source "./**/*.{vue,js,ts,jsx,tsx,wxml,axml,ttml,mpx}";
```

旧写法：

```ts title="vite.config.ts"
import tailwindcss from '@tailwindcss/vite'
import { defineConfig } from 'vite'
import { WeappTailwindcss } from 'weapp-tailwindcss/vite'

export default defineConfig({
  plugins: [
    tailwindcss(),
    WeappTailwindcss(),
  ],
})
```

v5 小程序推荐写法：

```ts title="vite.config.ts"
import path from 'node:path'
import { defineConfig } from 'vite'
import { WeappTailwindcss } from 'weapp-tailwindcss/vite'

export default defineConfig({
  plugins: [
    WeappTailwindcss({
      cssEntries: [
        path.resolve(__dirname, 'src/app.css'),
      ],
    }),
  ],
})
```

> 注意：不要把 `@import "tailwindcss"` 写进 `.scss`、`.less`、`.sass` 入口，也不要把 `cssEntries` 指向这些预处理文件。Tailwind CSS 4 官方也不建议和 Sass / Less / Stylus 组合使用。

## 7. 多入口和独立分包迁移

如果 v4 项目通过多个 `postcss.config.js` 上下文区分主包和独立分包，v5 迁移时改为多个 CSS 入口：

```ts title="vite.config.ts"
import path from 'node:path'
import { WeappTailwindcss } from 'weapp-tailwindcss/vite'

WeappTailwindcss({
  cssEntries: [
    path.resolve(__dirname, 'src/app.css'),
    path.resolve(__dirname, 'src/moduleA/app.css'),
  ],
})
```

每个入口可以使用自己的配置：

```css title="src/moduleA/app.css"
@config "../../tailwind.config.module-a.js";
@import "tailwindcss";
@source "./**/*.{vue,js,ts,jsx,tsx,wxml}";
```

这样主包和独立分包仍然有独立的 Tailwind 上下文，但生成、模板转译和 JS 转译都由 `WeappTailwindcss` 统一管理。

## 8. JS 类名和运行时合并迁移

v5 仍然按 `classSet` 精确命中模板和 JS 类名，不会启发式转译任意普通字符串。迁移后如果 `twMerge`、`twJoin`、`cva`、`tv`、`clsx` 这类运行时合并函数里出现类名，需要确认调用名被纳入识别范围。

```ts title="vite.config.ts"
WeappTailwindcss({
  cssEntries: [
    path.resolve(__dirname, 'src/app.css'),
  ],
  ignoreCallExpressionIdentifiers: [
    'twMerge',
    'twJoin',
    'cva',
    'tv',
    'clsx',
  ],
})
```

如果类名来自接口、数据库、远程配置或运行时拼接，Tailwind 本身也无法在构建期扫描到。迁移时应改成显式 safelist、`@source inline()` 或静态候选。

## 9. v5 配置变更对照

| v4 写法 | v5 写法 | 迁移说明 |
| --- | --- | --- |
| `postinstall: "weapp-tw patch"` | 删除 | 生成模式不再依赖安装后 patch |
| `tailwindcss-patch` | 删除 | v5 运行时生成 classSet |
| `postcss.config.js` 中的 `tailwindcss` | 删除 | Tailwind CSS 3.x 也由 `WeappTailwindcss` 生成 |
| 小程序构建中的 `@tailwindcss/postcss` | 删除 | 不再同时运行两套 Tailwind 生成链路 |
| 小程序构建中的 `@tailwindcss/vite` | 删除 | Vite 小程序构建只注册 `weapp-tailwindcss/vite` |
| `generator: false` | 删除 | v5 默认就是生成模式 |
| `mode` / `generator.mode` | 删除 | v5 不再提供旧后处理回退链路 |
| `target: 'weapp'` | 通常删除 | 小程序输出默认就是 `weapp` |
| `UnifiedViteWeappTailwindcssPlugin` | `WeappTailwindcss` | 使用当前公开插件名 |
| `@import "weapp-tailwindcss/index.css"` | `@import "tailwindcss"` | Tailwind CSS 4 入口使用官方 CSS-first 写法 |

## 10. 迁移后验证

在业务项目中至少验证一次对应端构建：

```bash
pnpm build
```

如果仓库内使用当前 monorepo 的 demo 或 app，可以按需运行：

```bash
pnpm build:apps
pnpm build:demo
pnpm --filter weapp-tailwindcss test
```

重点检查这些产物：

| 检查项 | 预期结果 |
| --- | --- |
| `app.wxss` / `app.acss` / `app.ttss` | 已输出小程序转义后的选择器 |
| `p-[20rpx]`、`text-[45rpx]` | 按合法长度单位生成，不被误判为颜色 |
| `hover:`、`dark:`、`group:` 等变体 | 小程序端输出符合当前框架能力 |
| `twMerge()` / `cva()` 中的类名 | 在构建产物中被正确转义 |
| `node_modules/.cache` | 不再出现旧 patch 刷新日志 |

## 11. 常见问题

**Q1: v5 是否必须同时升级到 Tailwind CSS 4?**

不必须。`weapp-tailwindcss@5` 同时支持 Tailwind CSS 3.x 和 4.x。需要兼容更多老设备时，可以继续使用 `tailwindcss@3`。

**Q2: H5 构建还能用 `@tailwindcss/vite` 吗?**

可以。H5 / Web 构建仍可使用官方 Tailwind 插件。限制只针对小程序构建：同一个小程序构建里不要同时注册官方 Tailwind 生成插件和 `WeappTailwindcss`。

**Q3: 迁移后样式没有生成，先查哪里?**

先查 `cssEntries` 是否指向真实存在的纯 `.css` 文件。Tailwind CSS 3.x 再查 `content`，Tailwind CSS 4.x 再查 `@source`。

**Q4: 迁移后 JS 字符串类名没有转义，是否应该启用启发式匹配?**

不应该。v5 继续坚持 `classSet` 精确命中。请把候选类写进 Tailwind 可扫描范围，或把运行时合并函数加入 `ignoreCallExpressionIdentifiers`。
