# 从 v4 迁移到 v5

`weapp-tailwindcss@5` 推荐让 `weapp-tailwindcss` 成为项目里唯一负责 Tailwind CSS 样式生成的入口。生成模式现在默认开启，常规小程序项目不需要再写 `generator: true`。

## 核心变化

1. 生成模式默认开启，默认配置等价于 `{ mode: 'auto', target: 'weapp' }`。
2. `mode: 'auto'` 会接管 Tailwind CSS v4 的样式生成，失败时回退到旧的 CSS 后处理链路。
3. Vite 入口默认会移除 `@tailwindcss/vite` 和 `@tailwindcss/postcss`，避免两套 Tailwind 生成链路同时运行。
4. CSS 入口必须是纯 `.css` 文件，不要把 `@import "tailwindcss"` 或 `cssEntries` 指向 `scss`、`less`、`sass`。
5. JS / 模板类名仍按 `classSet` 精确命中转译，不会启发式转译普通字符串。

## 依赖升级

```bash
pnpm add -D weapp-tailwindcss@5 tailwindcss@4
```

如果项目暂时还使用 `tailwindcss@3`，也可以只升级 `weapp-tailwindcss`：

```bash
pnpm add -D weapp-tailwindcss@5 tailwindcss@3
```

`tailwindcss@4` 项目使用 CSS-first 入口和 `@source`，会默认走 v5 生成模式；`tailwindcss@3` 项目继续使用 `tailwind.config.js` 的 `content`，默认保持旧的后处理兼容链路。

## 配置迁移

### Vite / weapp-vite / uni-app Vite

迁移后只保留 `weapp-tailwindcss/vite`，移除 `@tailwindcss/vite`：

```ts title="vite.config.ts"
import path from 'node:path'
import { defineConfig } from 'vite'
import { WeappTailwindcss } from 'weapp-tailwindcss/vite'

export default defineConfig({
  plugins: [
    WeappTailwindcss({
      rem2rpx: true,
      cssEntries: [
        path.resolve(__dirname, 'src/main.css'),
      ],
    }),
  ],
})
```

### Taro Vite

```ts title="config/index.ts"
import path from 'node:path'
import { defineConfig } from '@tarojs/cli'
import { WeappTailwindcss } from 'weapp-tailwindcss/vite'

export default defineConfig({
  compiler: {
    type: 'vite',
    vitePlugins: [
      WeappTailwindcss({
        rem2rpx: true,
        cssEntries: [
          path.resolve(__dirname, '../src/app.css'),
        ],
      }),
    ],
  },
})
```

### Webpack

Webpack 项目继续注册 `weapp-tailwindcss/webpack`，但不需要显式写 `generator: true`：

```js title="vue.config.js"
const path = require('node:path')
const { WeappTailwindcss } = require('weapp-tailwindcss/webpack')

module.exports = {
  configureWebpack(config) {
    config.plugins.push(
      new WeappTailwindcss({
        rem2rpx: true,
        cssEntries: [
          path.resolve(__dirname, './src/app.css'),
        ],
      }),
    )
  },
}
```

## CSS 入口

Tailwind CSS v4：

```css title="src/main.css"
@import "tailwindcss";
@source "./**/*.{vue,js,ts,wxml,tsx,jsx}";
@source not "../dist";
@source not "../unpackage";
```

Tailwind CSS v3：

```js title="tailwind.config.js"
export default {
  content: ['./src/**/*.{vue,js,ts,wxml,tsx,jsx}'],
  theme: {
    extend: {},
  },
}
```

## 兼容旧链路

如果项目暂时仍需要沿用 v4 的“先由官方 Tailwind 插件生成浏览器 CSS，再由 weapp-tailwindcss 后处理”的链路，可以显式关闭生成模式：

```ts
WeappTailwindcss({
  generator: false,
})
```

如果希望 CI 中严格校验生成模式，不允许失败后回退，可以使用：

```ts
WeappTailwindcss({
  generator: {
    mode: 'force',
  },
})
```

## 检查清单

1. 移除项目中的 `@tailwindcss/vite`、`@tailwindcss/postcss` 或 `tailwindcss` PostCSS 插件注册。
2. 确认 `cssEntries` 指向纯 `.css` 入口文件。
3. Tailwind CSS v4 项目确认 `@source` 覆盖模板和脚本文件。
4. 使用 `twMerge`、`twJoin`、`cva`、`tv` 时，确认 `ignoreCallExpressionIdentifiers` 覆盖了实际调用名。
5. 验证基础工具类、任意值（含 `rpx`）和变体类都能在目标小程序端生效。

## 推荐验证命令

```bash
pnpm --filter weapp-tailwindcss test
pnpm build:apps
pnpm build:demo
```

业务项目中至少运行一次对应端构建，并检查生成的 `app.wxss` / `app.acss` / `app.ttss` 中是否已经输出小程序转义后的选择器。
