# 从 v4 迁移到 v5

`weapp-tailwindcss@5` 的生成模式默认开启，`tailwindcss@3` 和 `tailwindcss@4` 项目都会默认由 `weapp-tailwindcss` 接管 Tailwind CSS 样式生成。迁移时仍然必须先判断 Tailwind 主版本，因为两个版本的扫描配置和 CSS 入口写法不同。

## 核心变化

1. 生成模式默认开启，默认配置等价于 `{ mode: 'auto', target: 'weapp' }`。
2. `tailwindcss@3` 和 `tailwindcss@4` 项目都会由 `weapp-tailwindcss` 接管样式生成，失败时直接让构建失败。
3. 大部分 Vite / Webpack 项目只保留 `weapp-tailwindcss/vite` 或 `weapp-tailwindcss/webpack` 的 `WeappTailwindcss` 插件，不再注册 `@tailwindcss/vite`、`@tailwindcss/postcss` 或 `tailwindcss` PostCSS 插件，避免两套 Tailwind 生成链路同时运行。
4. CSS 入口必须是纯 `.css` 文件，不要把 Tailwind 入口或 `cssEntries` 指向 `scss`、`less`、`sass`。
5. 生成模式项目不需要再在 `package.json` 里配置 `postinstall: "weapp-tw patch"`。
6. Tailwind CSS v3 继续使用 `tailwind.config.js` 的 `content`，Tailwind CSS v4 继续使用 CSS 入口里的 `@source`。
7. JS / 模板类名仍按 `classSet` 精确命中转译，不会启发式转译普通字符串。

## 迁移路径选择

| 当前项目 | 推荐路径 | Tailwind 配置 | 官方 Tailwind 插件 | 改造成本 |
| --- | --- | --- | --- | --- |
| 已经使用 `tailwindcss@4` | 升级到 `weapp-tailwindcss@5`，使用默认生成模式 | CSS-first，使用 `@source` | 不再注册 `@tailwindcss/vite` / `@tailwindcss/postcss` | 需要整理 CSS 入口 |
| 仍在使用 `tailwindcss@3` | 升级到 `weapp-tailwindcss@5`，使用默认生成模式 | `tailwind.config.js` 的 `content` | 不再注册 `tailwindcss` PostCSS 插件 | 需要把 CSS 入口交给 `cssEntries` |

## 依赖升级

```bash
pnpm add -D weapp-tailwindcss@5 tailwindcss@4
```

如果项目暂时还使用 `tailwindcss@3`，推荐只升级 `weapp-tailwindcss`：

```bash
pnpm add -D weapp-tailwindcss@5 tailwindcss@3
```

`tailwindcss@4` 项目使用 CSS-first 入口和 `@source`，`tailwindcss@3` 项目继续使用 `tailwind.config.js` 的 `content`。两者都会默认走 v5 直出生成。

## Tailwind CSS v4 配置迁移

Tailwind CSS v4 项目迁移后只保留 `weapp-tailwindcss` 的构建器入口，让它成为唯一的 Tailwind CSS 生成入口。

迁移后的默认形态是：Vite 项目只注册 `weapp-tailwindcss/vite` 的 `WeappTailwindcss`，Webpack 项目只注册 `weapp-tailwindcss/webpack` 的 `WeappTailwindcss`。如果项目里原来有 `@tailwindcss/vite`、`@tailwindcss/postcss` 或 `tailwindcss` PostCSS 插件，把这些 Tailwind 官方生成插件移除；项目自己的非 Tailwind PostCSS 插件可以继续保留。

### Vite / weapp-vite / uni-app Vite

移除 `@tailwindcss/vite` 和 `@tailwindcss/postcss`，只保留 `WeappTailwindcss`：

```ts title="vite.config.ts"
import path from 'node:path'
import { defineConfig } from 'vite'
import { WeappTailwindcss } from 'weapp-tailwindcss/vite'

export default defineConfig({
  plugins: [
    WeappTailwindcss({
      rem2rpx: true,
      cssEntries: [
        path.resolve(__dirname, 'src/main.css'),
      ],
    }),
  ],
})
```

### Taro Vite

```ts title="config/index.ts"
import path from 'node:path'
import { defineConfig } from '@tarojs/cli'
import { WeappTailwindcss } from 'weapp-tailwindcss/vite'

export default defineConfig({
  compiler: {
    type: 'vite',
    vitePlugins: [
      WeappTailwindcss({
        rem2rpx: true,
        cssEntries: [
          path.resolve(__dirname, '../src/app.css'),
        ],
      }),
    ],
  },
})
```

### Webpack

Webpack 项目继续注册 `weapp-tailwindcss/webpack`，但不需要显式写 `generator: true`：

```js title="vue.config.js"
const path = require('node:path')
const { WeappTailwindcss } = require('weapp-tailwindcss/webpack')

module.exports = {
  configureWebpack(config) {
    config.plugins.push(
      new WeappTailwindcss({
        rem2rpx: true,
        cssEntries: [
          path.resolve(__dirname, './src/app.css'),
        ],
      }),
    )
  },
}
```

Tailwind CSS v4：

```css title="src/main.css"
@import "tailwindcss";
@source "./**/*.{vue,js,ts,wxml,tsx,jsx}";
@source not "../dist";
@source not "../unpackage";
```

## Tailwind CSS v3 配置迁移

Tailwind CSS v3 项目不需要改成 CSS-first，但需要让 `weapp-tailwindcss` 成为 Tailwind 样式生成入口。迁移后移除原来的 `tailwindcss` PostCSS 插件，扫描范围仍然写在 `tailwind.config.js` 的 `content`。也就是说，v3 生成模式下只需要安装并注册 `WeappTailwindcss` 构建器插件，不需要把 `tailwindcss` 再写进 `postcss.config.js`。

```ts title="vite.config.ts"
import path from 'node:path'
import { defineConfig } from 'vite'
import { WeappTailwindcss } from 'weapp-tailwindcss/vite'

export default defineConfig({
  plugins: [
    WeappTailwindcss({
      rem2rpx: true,
      cssEntries: [
        path.resolve(__dirname, 'src/main.css'),
      ],
    }),
  ],
})
```

```css title="src/main.css"
@tailwind base;
@tailwind components;
@tailwind utilities;
```

```js title="tailwind.config.js"
export default {
  content: ['./src/**/*.{vue,js,ts,wxml,tsx,jsx}'],
  theme: {
    extend: {},
  },
}
```

## 生成模式是唯一链路

v5 不再保留“先由官方 Tailwind 插件生成浏览器 CSS，再由 weapp-tailwindcss 后处理”的旧链路。历史配置里的 `generator: false` 会按默认生成模式处理；项目中应移除官方 `@tailwindcss/vite`、`@tailwindcss/postcss` 或 `tailwindcss` PostCSS 插件注册。

## 迁移稳定期废弃配置

v5 进入默认生成模式后，下面这些配置或写法只保留兼容旧项目。新项目和完成迁移后的项目应删除这些配置，减少两套链路混用带来的排障成本。

| 废弃配置或写法 | 替代方式 | 说明 |
| --- | --- | --- |
| `generator: false` | 删除该配置 | v5 已移除旧的“只后处理 CSS”链路，`false` 会按默认生成模式处理。 |
| `generator: true` | 删除该配置 | 默认已经启用生成模式，显式传 `true` 没有额外效果。 |
| `generator: { mode: 'force' }` | 删除 `mode` | 默认生成模式已经是稳定链路；仅历史测试或排障代码可能还会保留该写法。 |
| `generator: { target: 'weapp' }` | 删除 `target` | 小程序构建默认就是 `weapp`；只有调试 `web` / `tailwind` 目标时才需要显式配置。 |
| PostCSS 插件顶层 `target` | 改为 `generator.target` | 顶层 `target` 是兼容旧 PostCSS 入口的别名。 |
| `staleClassNameFallback` | 使用 `jsArbitraryValueFallback` | JS 转译已经统一按 `classNameSet` 精确命中，旧兜底字段不再参与候选判定。 |
| `rewriteCssImports` | 运行时 CSS 入口直接写 `@import "weapp-tailwindcss/index.css"` | 该配置保留给旧项目和 IntelliSense 迁移场景，新项目不建议依赖导入重写。 |
| `UnifiedViteWeappTailwindcssPlugin` | `WeappTailwindcss` | 旧导出名继续兼容，文档和业务配置统一使用推荐别名。 |

## 检查清单

1. 移除项目中的 `@tailwindcss/vite`、`@tailwindcss/postcss` 或 `tailwindcss` PostCSS 插件注册。
2. 移除 `package.json` 中的 `postinstall: "weapp-tw patch"`。
3. 确认 `cssEntries` 指向纯 `.css` 入口文件。
4. Tailwind CSS v3 项目确认 `tailwind.config.js content` 覆盖模板和脚本文件。
5. Tailwind CSS v4 项目确认 `@source` 覆盖模板和脚本文件。
6. 使用 `twMerge`、`twJoin`、`cva`、`tv` 时，确认 `ignoreCallExpressionIdentifiers` 覆盖了实际调用名。
7. 删除迁移稳定期废弃配置，确认业务配置只保留默认生成模式需要的 `WeappTailwindcss`、`cssEntries` 和样式转换选项。
8. 验证基础工具类、任意值（含 `rpx`）和变体类都能在目标小程序端生效。

## 推荐验证命令

```bash
pnpm --filter weapp-tailwindcss test
pnpm build:apps
pnpm build:demo
```

业务项目中至少运行一次对应端构建，并检查生成的 `app.wxss` / `app.acss` / `app.ttss` 中是否已经输出小程序转义后的选择器。
