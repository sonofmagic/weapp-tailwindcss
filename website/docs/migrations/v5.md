# 从 v4 迁移到 v5

v5 最大的变化只有一句话：

**小程序端的 Tailwind CSS 生成，默认交给 `WeappTailwindcss` 来做。**

所以迁移时不要想太复杂。大多数项目只需要做这几件事：

1. 升级 `weapp-tailwindcss`
2. 删除旧的 `weapp-tw patch`
3. 不再给小程序构建注册 Tailwind 官方 Vite / PostCSS 插件
4. 注册 `WeappTailwindcss`
5. 跑一次构建，看样式有没有出来

## 1. 先看你用的是 Tailwind CSS 3 还是 4

这一步很重要，因为两者入口不一样。

| 你现在用的版本 | CSS 入口怎么写 | 扫描范围写在哪里 |
| --- | --- | --- |
| `tailwindcss@3` | `@tailwind base;` / `@tailwind components;` / `@tailwind utilities;` | `tailwind.config.js` 的 `content` |
| `tailwindcss@4` | `@import "tailwindcss";` | CSS 里的 `@source` |

升级到 `weapp-tailwindcss@5` 不要求你同时升级到 `tailwindcss@4`。

如果你的项目要兼容更多老设备，继续用 `tailwindcss@3` 没问题。

## 2. 升级依赖

继续使用 Tailwind CSS 3：

```bash
pnpm add -D weapp-tailwindcss@5 tailwindcss@3
```

使用 Tailwind CSS 4：

```bash
pnpm add -D weapp-tailwindcss@5 tailwindcss@4
```

如果小程序构建里以前装了这些包，通常可以删掉：

```bash
pnpm remove tailwindcss-patch @tailwindcss/vite @tailwindcss/postcss
```

如果你的 H5 / Web 构建还在用 `@tailwindcss/vite` 或 `@tailwindcss/postcss`，那就别删。只要保证**小程序构建**不要同时跑官方 Tailwind 插件和 `WeappTailwindcss`。

## 3. 删除旧 patch 脚本

打开 `package.json`，删掉这一行：

```jsonc title="package.json"
{
  "scripts": {
    // 删除
    "postinstall": "weapp-tw patch"
  }
}
```

v5 不需要安装后 patch。还保留这行的话，后面排查问题会很绕。

## 4. 注册 WeappTailwindcss

迁移时通常不用手动配置 `cssEntries`。

只要你的 Tailwind 入口 CSS 被框架正常引入，`weapp-tailwindcss@5` 会在构建过程中自动识别它。

很多项目本来就有 `src/app.css`、`src/main.css` 或 `app.css`。你只要确认这个文件里有 Tailwind 入口即可。

如果项目里还没有这样的文件，再新建一个纯 `.css` 文件即可。

Tailwind CSS 3 项目入口通常是这样：

```css title="src/app.css"
@tailwind base;
@tailwind components;
@tailwind utilities;
```

Tailwind CSS 4 项目入口通常是这样：

```css title="src/app.css"
@import "tailwindcss";
@source "./**/*.{vue,js,ts,jsx,tsx,wxml,axml,ttml,mpx}";
@source not "../dist";
@source not "../unpackage";
@source not "../node_modules";
```

Tailwind CSS 4 的入口不要直接写进 `scss`、`less`、`sass` 文件。先放在纯 CSS 里，业务样式要用预处理器时再去引入它。

### Vite / uni-app Vite / weapp-vite

```ts title="vite.config.ts"
import path from 'node:path'
import { defineConfig } from 'vite'
import { WeappTailwindcss } from 'weapp-tailwindcss/vite'

export default defineConfig({
  plugins: [
    WeappTailwindcss({
      rem2rpx: true,
    }),
  ],
})
```

`uni-app Vite` 项目里，放在 `uni()` 后面：

```ts title="vite.config.ts"
import path from 'node:path'
import uni from '@dcloudio/vite-plugin-uni'
import { defineConfig } from 'vite'
import { WeappTailwindcss } from 'weapp-tailwindcss/vite'

export default defineConfig({
  plugins: [
    uni(),
    WeappTailwindcss({
      rem2rpx: true,
    }),
  ],
})
```

### Webpack / Taro Webpack / Mpx

普通 Webpack 写法：

```js title="webpack.config.js"
const path = require('node:path')
const { WeappTailwindcss } = require('weapp-tailwindcss/webpack')

module.exports = {
  plugins: [
    new WeappTailwindcss({
      rem2rpx: true,
    }),
  ],
}
```

Taro Webpack 通常写在 `mini.webpackChain`：

```js title="config/index.js"
const path = require('node:path')
const { WeappTailwindcss } = require('weapp-tailwindcss/webpack')

module.exports = {
  mini: {
    webpackChain(chain) {
      chain.merge({
        plugin: {
          install: {
            plugin: WeappTailwindcss,
            args: [{
              rem2rpx: true,
            }],
          },
        },
      })
    },
  },
}
```

### 什么时候还要手动配 cssEntries？

大多数项目不用配。下面这些情况再手动写：

- 你有多个 Tailwind CSS 入口
- 你的入口 CSS 没有被框架正常引入
- 你在自定义构建脚本里调用 `weapp-tailwindcss`
- 自动识别失败，构建日志提示需要指定入口

写法也很简单：

```ts title="vite.config.ts"
import path from 'node:path'
import { WeappTailwindcss } from 'weapp-tailwindcss/vite'

WeappTailwindcss({
  cssEntries: [
    path.resolve(__dirname, 'src/app.css'),
  ],
})
```

## 5. 从 PostCSS 里拿掉 Tailwind

如果你的小程序构建里还有这种配置：

```js title="postcss.config.js"
module.exports = {
  plugins: {
    tailwindcss: {},
    autoprefixer: {},
  },
}
```

迁移后改成只保留业务真的需要的 PostCSS 插件：

```js title="postcss.config.js"
module.exports = {
  plugins: {
    // 这里不要再放 tailwindcss 或 @tailwindcss/postcss
  },
}
```

简单说：小程序端不要跑两套 Tailwind 生成链路。

## 6. 跑一次构建

运行你的项目构建命令：

```bash
pnpm build
```

或者对应框架的开发命令，例如：

```bash
pnpm dev:mp-weixin
```

然后随便找个页面写一个明显的类：

```html
<view class="mt-4 rounded bg-sky-500 p-4 text-white">Hello</view>
```

能看到样式，说明主链路基本通了。

## 7. 出问题先看这几个地方

| 现象 | 先检查什么 |
| --- | --- |
| 完全没有 Tailwind 样式 | Tailwind 入口 CSS 是否被框架正常引入 |
| Tailwind CSS 3 类名没生成 | `tailwind.config.js` 的 `content` 是否扫到了源码 |
| Tailwind CSS 4 类名没生成 | `src/app.css` 的 `@source` 是否扫到了源码 |
| 构建里有奇怪重复样式 | 是否还注册了 `tailwindcss` / `@tailwindcss/postcss` / `@tailwindcss/vite` |
| 安装后还在跑 patch | `package.json` 里是否还留着 `postinstall: "weapp-tw patch"` |
| JS 里的类名没有转义 | 这个类名是否能被 Tailwind 扫描到 |

## 8. 多入口项目怎么迁移

如果你有主包、独立分包，或者多个 Tailwind CSS 入口，再把它们写进 `cssEntries`：

```ts title="vite.config.ts"
import path from 'node:path'
import { WeappTailwindcss } from 'weapp-tailwindcss/vite'

WeappTailwindcss({
  cssEntries: [
    path.resolve(__dirname, 'src/app.css'),
    path.resolve(__dirname, 'src/module-a/app.css'),
  ],
})
```

每个入口还是按自己的 Tailwind 版本写：

- Tailwind CSS 3：用 `@tailwind` 和 `tailwind.config.js`
- Tailwind CSS 4：用 `@import "tailwindcss"` 和 `@source`

## 9. 你可能会问

### 一定要升级 Tailwind CSS 4 吗？

不一定。`weapp-tailwindcss@5` 支持 Tailwind CSS 3 和 4。

想稳一点，就先继续用 `tailwindcss@3`。以后要用 Tailwind CSS 4，再单独迁移。

### H5 还能用 Tailwind 官方插件吗？

可以。H5 / Web 构建想怎么用 Tailwind 官方插件都可以。

这里说的限制只针对小程序构建：同一个小程序构建里，不要同时注册官方 Tailwind 生成插件和 `WeappTailwindcss`。

### v5 还会自动转 JS 字符串里的类名吗？

会，但前提是这些类名先被 Tailwind 扫描到了。

`weapp-tailwindcss` 不会猜普通字符串是不是类名。它只处理 Tailwind 已经确认过的 classSet，这样更稳，也更不容易误伤业务字符串。
