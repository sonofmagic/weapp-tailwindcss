# 逐段口播稿

## 1. 开场 3 分钟

大家晚上好，今天这场直播的题目叫“小程序还能这么写？AI 出样式，Tailwind 跑全端”。

我先问大家一个很现实的问题。你平时做小程序开发，到底有多少时间花在业务逻辑上，又有多少时间花在调 `margin`、调 `padding`、调圆角、调阴影、调字号？很多时候我们嘴上说是在做前端，实际上是在做像素搬运工。

尤其是小程序场景，很多同学到了 2026 年，还是在手写 `margin-left: 20rpx`、`padding: 24rpx 32rpx`，然后改一处样式，切一次设计稿，切一次 IDE，再切一次开发者工具。效率低，反馈慢，还很容易出现多端不一致。

所以今天我想讲的不是“怎么再学一种 CSS 写法”，而是换一个范式。我们把样式从“体力活”变成“表达问题”。一句话描述 UI，让 AI 先生成 Tailwind 原子类，然后交给 `weapp-tailwindcss` 转成小程序能跑的结果，最后直接预览。今天我尽量挑战一件事，全程少手写甚至不手写传统 CSS。

## 2. 痛点铺垫 5 分钟

为什么小程序样式开发一直痛？我觉得有三个原因。

第一，表达成本高。设计稿给你的是视觉结果，但你实现时要自己拆成字号、间距、圆角、背景、边框、布局、状态。整个过程非常机械。

第二，试错成本高。你脑子里觉得“这里应该再轻一点、再柔一点、再有层次一点”，但真正要落下去，得改很多细碎属性。

第三，跨端一致性差。你做 `uni-app`、`taro`、原生、甚至 `uni-app x` 的时候，不同工具链、不同端的样式适配问题会把人拖回到非常底层的细节里。

所以我的核心观点是，样式如果还是以“手写属性”为中心，就很难提效。但如果样式被表达成一种带结构、带约束、可枚举的语言，AI 就能介入。Tailwind 恰好就是这种语言。

## 3. 新范式演示 7 分钟

我先不给大家讲原理，我们先看结果。

现在我对 AI 说一句话：“做一个渐变卡片，顶部是标题和副标题，卡片圆角大一点，阴影柔和一点，右上角有一个轻微高光，底部放一个醒目的按钮，整体偏科技感，但不要太炫。”  
如果这句话拿去让 AI 直接生成手写 CSS，它常常会输出一大段看起来很完整、但你未必敢直接落地的代码。  
但如果目标是 Tailwind 类名，它会稳定很多。因为它输出的不是自由发挥的视觉散文，而是一串有规则的原子类。

接下来我把这串类名直接放到页面里。这里最关键的不是“AI 写得有多美”，而是这串类名进入我们的工程链路以后，`weapp-tailwindcss` 会自动帮我们做小程序适配。也就是说，AI 负责样式意图表达，插件负责小程序翻译，开发者负责验收和微调。

这一步一旦通了，整个开发感受就变了。你不再是先想 CSS 属性，而是先想“我要什么界面语义”，然后让 AI 用 Tailwind 说出来。

## 4. 为什么 AI + Tailwind 是天作之合 8 分钟

很多人会问，为什么不是 AI + SCSS，为什么偏偏是 AI + Tailwind？

原因很简单。Tailwind 的类名，本质上是一种半结构化语言。  
它有明确词汇表，有稳定组合方式，有足够强的约束。  
例如 `rounded-2xl`、`shadow-lg`、`px-4`、`text-[28rpx]`、`bg-linear-to-r`、`from-cyan-500`、`to-blue-600`，这些东西既可读，又可枚举。

这正是 AI 最擅长的领域。AI 不擅长没有边界的创作，但很擅长“在一个有限语法空间里做高质量组合”。

所以你会发现一个很有意思的现象。  
让 AI 生成一整份传统 CSS，容易出现冗长、重复、结构松散。  
但让 AI 生成一段 Tailwind 类名，它往往又快又准，因为原子类本身就是一种受限语言。

这就是为什么今天的主角不是 AI，也不是 Tailwind，而是这两者加起来，再通过 `weapp-tailwindcss` 真正落到小程序。

## 5. 项目全景 8 分钟

现在我们拉回到这个仓库本身。

这个项目最值得讲的一点，是它已经不是“一个小插件”。仓库里可以直接看到，它在 README 里明确写了支持 `webpack`、`vite`、`rspack`、`rollup`、`rolldown`、`gulp` 这些构建工具基底。也就是说，它不是绑死在某一个框架里的。

第二，仓库里有大量 demo 和模板。你能看到 `uni-app`、`taro`、`mpx`、原生小程序、`weapp-vite`，还有 `uni-app x` 的相关演示。也就是说，这不是口头上的“理论兼容”，而是仓库里确实放了样例工程。

第三，它已经把 AI 使用方式沉淀成了 Skill。这个特别重要。因为很多人说“让 AI 帮你做配置”，但没有 workflow。这个项目不一样，它把 AI 该先问什么、该收集什么信息、该输出什么结果，都写进了 `SKILL.md`。

第四，它还有 watch/HMR 回归链路和 benchmark。说明作者在关心开发效率，而不只是一次 build 成功。

所以你今天看到的是一个很完整的工程思路：  
不是“Tailwind 能不能跑在小程序”，而是“AI + Tailwind + 小程序 + 多框架 + 多端 + 验证链路”能不能成为真正可用的日常方案。

## 6. 实操 Round 1：uni-app + Tailwind CSS v4 12 分钟

下面我们做第一轮实操，目标非常简单：5 分钟从零到页面。

我选择的示例是仓库里的 `demo/uni-app-tailwindcss-v4`。为什么选它？因为它足够直观，也足够贴近大多数团队。它用的是 `uni-app cli vue3 vite` 这条路线，认知成本低。

先看三个地方。

第一，看 `package.json`。这里有一行特别关键：`postinstall: "weapp-tw patch"`。这行不能漏。很多同学样式不生效、任意值异常、JS 里的 class 不工作，根上往往都是 patch 链路没打通。

第二，看 `vite.config.ts`。这里注册了 `uni()`、Tailwind 插件，以及 `UnifiedViteWeappTailwindcssPlugin`。这意味着 Tailwind 负责生成原子类，`weapp-tailwindcss` 负责把结果改造成适合小程序环境的样子。

第三，看 `src/main.css`。这里不是在写传统样式，而是在定义 Tailwind 入口、主题变量和配置入口。

接下来我在页面里直接写类名。不是写 `.card {}`，也不是开一份 SCSS，而是直接把结构和样式意图写在模板里。  
这一步如果配合 AI 来做，速度会更快。  
我只要说“给我一个带渐变背景、圆角、柔和阴影、按钮高亮的卡片”，AI 返回一串类名，我贴进去就能看结果。

这就是今天希望大家带走的第一件事。`weapp-tailwindcss` 不是让你多学一堆配置，而是帮你把“样式表达”从传统 CSS 切换到更适合 AI 的语言体系。

## 7. 实操 Round 2：AI Skill 驱动全流程 13 分钟

第二轮，我想展示的不是“AI 会写代码”，而是“AI 会按流程做工程决策”。

我们看仓库里的 Skill 文档。这里面不是一句“请帮我接入 Tailwind”，而是明确要求先收集最小上下文：

- 你是什么框架
- 你用什么构建器
- 目标端是什么
- Tailwind 是 v3 还是 v4
- 你是不是 `pnpm@10+`

为什么这很重要？因为 AI 最怕上下文不完整。  
一旦上下文不完整，它就会给你一个听起来都对、但落地时四处漏风的答案。  
Skill 的价值，就是先把 AI 约束住。

然后它的输出也不是泛泛而谈，而是要求必须包含：

- 修改文件清单
- 可复制配置
- 安装命令
- 验证步骤
- 回滚方案

这就把“AI 帮你写代码”升级成“AI 帮你走工作流”了。

如果今天你在公司里推这套方案，我非常建议你讲这一点。因为团队最担心的不是 AI 会不会生成页面，而是它有没有工程可靠性。  
有 Skill，你就可以说，我们不是让 AI 自由发挥，而是让它按清单执行。

## 8. 进阶技巧 8 分钟

讲完主链路，必须补几个最容易踩坑的地方。

第一，任意值和 `rpx`。  
像 `text-[22rpx]` 这种写法在小程序里很好用，但某些二义性场景需要更明确的类型前缀。比如 `text-[length:22rpx]`。所以 patch 链路和任意值规则一定要知道。

第二，动态 class。  
不要写 `bg-${color}-500` 这种半截拼接。对 AI 也一样，不要让它自由造 token。正确方式是枚举。让它在完整字面量里选。

第三，`space-y` 和 `space-x`。  
在小程序环境下，这类依赖子组合器的工具类不是天然全标签通用。仓库自己的最佳实践里给出的顺序很清楚：先改结构，再看 `virtualHost`，最后再扩展配置。

第四，运行时合并。  
如果你的组件库要做外部类名覆盖，`twMerge`、`cva`、`cn` 这种能力就很关键。它能让“默认样式 + 外部覆盖 + 冲突去重”这条链路更稳。

第五，`uni-app x`。  
README 已经明确提到，`weapp-tailwindcss@4.2.x` 支持 `uni-app x` 同时构建 `Web`、小程序、`Android`、`iOS`、鸿蒙。这一点很适合放在直播后半段讲，因为它能把“我只是给小程序接个 Tailwind”一下子抬升成“我在为多端统一样式语言做准备”。

## 9. 数据收尾 4 分钟

最后我想给大家看两个仓库里的工程信号。

一个是 2026 年 2 月 13 日的热更新报告。仓库专门做了 watch/HMR 的回归链路，这说明作者在盯“改一行类名以后我多久能看到结果”。

另一个是 2026 年 2 月 23 日生成的框架对比报告。在那份统一口径 benchmark 里，`weapp-vite wevu` 这组在 Build 和 HMR 中位数上更低。这里我不是要下绝对结论说谁一定最好，而是想说明，这个仓库已经开始把开发体验量化了。

这很重要。因为我们今天讲 AI，讲 Tailwind，讲小程序，如果最后只停在“能跑”，其实是不够的。真正能进团队、进生产的方案，一定还要回答效率、稳定性和可验证性。

## 10. 收尾与 CTA 2 分钟

所以今天这场直播，你如果只带走一句话，我希望是这句：

不要再把小程序样式开发理解成手写 CSS 的体力活了。  
把它变成一句可描述的意图，让 AI 用 Tailwind 说出来，再让 `weapp-tailwindcss` 帮你翻译到小程序和多端。

如果你回去想自己试，最简单的动作有三个：

1. 先跑一个仓库里的 demo
2. 再安装 `weapp-tailwindcss` Skill
3. 最后用一句真实业务需求，试一次“不手写 CSS”的页面生成

如果你觉得这条路值钱，记得给仓库点个 Star。因为这类项目真正难的不是一个功能点，而是把多框架、多端、AI、Tailwind、回归链路一整套揉在一起，还持续维护。
