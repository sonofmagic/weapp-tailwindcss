然后你想改哪里，| 直接改，| 热更新一下，| 马上就能看到结果。||

所以为什么这个题值得专门讲一场。||

因为很多人平时说自己在做前端，| 实际上大把时间都花在样式搬运上。||

尤其是小程序。||

你改一个间距，| 切一次设计稿。||

你改一个阴影，| 切一次 IDE。||

你再看结果，| 还得切一次开发者工具或者真机。||

最后一天过去了，| 业务没推进多少，| 样式倒是调了半天。||

这件事为什么慢。||

我觉得核心就三个字。||

>>> 不划算。||

第一，| 表达成本高。||

设计稿给你的是一个视觉结果。||

是太重了。||

太硬了。||

太平了。||

还是层次不够。||

于是就只能继续调。||

第三，| 反馈链路长。||

尤其在小程序里，| 还叠加了构建工具、运行端、框架差异、单位适配、动态 class、生效顺序这些问题。||

所以它不是简单的“写样式”。||

它是一整条很碎的反馈链路。||

所以我一直觉得，| 不是大家不会写。||

是我们还在用一种特别笨、特别慢的方式写。||

如果样式开发一直围绕手写底层属性，| AI 就很难稳定帮上忙。||

但如果样式本身是一种可枚举、可约束、可组合的语言，| AI 用起来就会顺很多。||

Tailwind，| 恰好就是这个中间层。||

## 样式方案演进

【操作】

切到：

`/docs/tailwindcss/history`

接下来我先讲历史。||

不是为了背时间线。||

而是为了说明一件事。||

Tailwind 不是凭空冒出来的。||

它接的是前面很多年一直没完全解决掉的工程问题。||

如果把前端样式方案粗一点看，| 差不多就是这么一路走过来的。||

最早是 Raw CSS。||

后来大家发现，| 只靠全局 CSS 很容易乱。||

于是有了 BEM、OOCSS 这种命名方法。||

再往后，| 大家想让 CSS 更像工程代码，| 就有了 Sass、Less 这些预处理器。||

然后大家又发现，| 光有变量、混入、嵌套还不够。||

难点其实是作用域。||

所以又有了 CSS Modules。||

再后来，| 组件越来越重要，| 动态状态越来越多。||

于是 CSS-in-JS 也兴起过。||

再到今天，| Utility-first 开始变成很多团队的落点。||

但这里有一句话一定要讲清楚。||

这不是线性替代史。||

不是说 Tailwind 出现了，| 前面的方案就都输了。||

现实一直是多方案并行。||

所以做技术选型的时候，| 最没意义的问题就是“谁赢了”。||

该问的是：| 什么场景下，| 谁更合适。||

### Raw CSS

【操作】

切到：

`/docs/tailwindcss/history/raw-css`

Raw CSS 的优点很明显。||

直接。||

简单。||

没门槛。||

小页面、小站点、低复杂度场景，| 它完全成立。||

但它的问题也同样明显。||

全局污染。||

覆盖链条不透明。||

命名冲突。||

项目一大，| 你就会越来越怕改样式。||

所以后来才有了 BEM、OOCSS。||

这些东西不是没价值。||

它们的价值很大。||

因为它们第一次很认真地把命名和结构当成工程手段。||

只是它们主要还是靠纪律。||

靠人脑。||

靠 review。||

这就决定了它们很难从根上解决复杂度。||

### 预处理器

【操作】

切到：

`/docs/tailwindcss/history/preprocessors`

预处理器当年为什么这么吸引人。||

因为它第一次让 CSS 明显更像工程代码。||

变量。||

混入。||

函数。||

嵌套。||

这些东西在今天看起来很正常。||

但在当时，| 确实是一波生产力升级。||

不过它解决的主要还是“写”的重复。||

没有完全解决“管”的复杂度。||

你有了更强的语法，| 不代表你就有了作用域。||

你有了嵌套，| 甚至还可能更容易掉进选择器地狱。||

### CSS Modules

【操作】

切到：

`/docs/tailwindcss/history/css-modules`

CSS Modules 很关键的一步是什么。||

就是作用域开始交给编译器。||

而不是继续靠命名纪律。||

这其实是一个很大的工程思路变化。||

因为只要一个问题能由编译器稳定保证，| 你就不该继续主要依赖人脑。||

所以 CSS Modules 一直到今天，| 对中大型应用和组件库都还是很有价值。||

它不花哨。||

但它稳。||

### CSS-in-JS

【操作】

切到：

`/docs/tailwindcss/history/css-in-js`

CSS-in-JS 当年为什么会火。||

一个很现实的原因就是，| 像 React 这种视图层，| 本来就已经在用 JS 和 JSX 了。||

那很多人自然会觉得，| 样式也放进 JS 里，| 好像很顺。||

再加上它确实把样式和状态绑得更紧。||

主题。||

props。||

运行时状态。||

都能更自然地进入样式表达。||

这对设计系统、多品牌、多主题，| 吸引力很大。||

但代价也是真的。||

运行时成本。||

SSR 注水。||

构建链复杂。||

调试也更重。||

所以后来很多团队又继续往前走。||

慢慢开始偏向编译期、偏向零运行时。||

本质上还是在找新的平衡点。||

### Utility-first

【操作】

切到：

`/docs/tailwindcss/history/utility-first`

最后就到了 Utility-first。||

也就是 Tailwind、UnoCSS 这一类。||

它们为什么会在今天成立。||

因为今天团队最头疼的，| 已经不是“样式写不写得出来”。||

而是“样式约束能不能稳定复用、组合、审阅、生成”。||

所以组织方式开始变。||

从命名 class，| 变成组合 class。||

从局部抽象，| 变成 tokens 和 variants。||

从运行时解释，| 变成构建期裁剪和生成。||

所以今天原子化 CSS 才会显得这么合理。||

不是因为大家突然不想写 CSS。||

而是因为今天的工程条件，| 刚好特别适合它。||

这一页记一句最简单的话就够了。||

今天很多成熟团队，| 已经不再只把样式问题看成“怎么写 class”。||

他们会把它拆成几层。||

tokens。||

variants。||

builder。||

交互 primitives。||

所以后面讲组件工程的时候，| 自然就会讲到 shadcn/ui、Headless、builder 这些东西。||

## Tailwind 到底是什么

【操作】

切到：

`/docs/tailwindcss/tailwind-core`

很多人第一次接触 Tailwind，| 会把它理解成“把 CSS 写进 className”。||

这个理解不算错。||

但这还是太浅了。||

Tailwind 真正的核心，| 不是 class 这个表面意思。||

而是它背后的整条链路。||

我一般把它拆成三段。||

第一段，| 入口。||

入口不是 class。||

入口是 token。||

颜色、间距、圆角、字号、阴影、动效，| 这些值最好先收敛。||

不然你本质上只是换一种方式继续裸写视觉值。||

第二段，| 中段。||

中段是生成机制。||

content 扫描。||

JIT 生成。||

`@layer` 管层级。||

plugins 扩能力。||

这一步的重点不是类名有多炫。||

而是它让样式变成一门更容易被构建工具消费的语言。||

第三段，| 出口。||

出口不是 utility 本身。||

出口是组件语义。||

也就是说，| 最后落到业务里的，| 不该是一坨越来越长的 className。||

而应该是有默认值、有变体、有边界的组件 API。||

所以 Tailwind 解决的，| 不只是少写 CSS。||

它更大的价值是：| 让设计约束、类名组合、组件语义和构建产物更容易对齐。||

你们如果只记这一页一句话。||

就记这句。||

Tailwind 不是类名表。||

它是一条工程流水线。||

## 为什么很多团队会用乱

【操作】

切到：

`/docs/tailwindcss/best-practices`

Tailwind 最大的风险是什么。||

不是难。||

恰恰是太容易开始。||

你装完几乎马上就能写页面。||

反馈也很快。||

于是很多团队直接跳过了中间建设。||

tokens 没抽。||

builder 没建。||

merge 没接。||

content 没收紧。||

评审规则也没立。||

这种情况下，| 前期当然很爽。||

但后面问题会慢慢全部回来。||

类名越来越长。||

关系类越来越深。||

状态散落在业务组件里。||

动态类开始字符串拼接。||

每一段代码都像“能跑”，| 但很难判断是不是长期正确。||

所以 Tailwind 最大的误区就是：| 因为它太快，| 大家误以为它不需要结构。||

其实正好相反。||

它非常需要结构。||

而且比传统 CSS 时代更需要。||

我觉得至少要把四件事立起来。||

第一，| tokens。||

别在业务组件里到处写裸色值、裸字号、裸间距。||

先有一份设计 token。||

这样以后改主题、改品牌、做暗色模式，| 改的是映射，| 不是全项目搜字符串。||

第二，| variants。||

只要组件有 `size`、`tone`、`state`、`intent` 这些维度，| 就别让它们散在业务组件里。||

集中到 builder 层。||

第三，| merge。||

这一点很关键。||

`tailwind-merge` 不是锦上添花。||

它解决的是类冲突和外部覆盖的可预测性。||

第四，| content 和构建验证。||

Tailwind 的很多问题根本不在运行时。||

它在构建时。||

扫描范围太宽。||

动态类太随意。||

关系类太深。||

都会让产物失控。||

所以成熟的 Tailwind 工程，| 不只是有 `tailwind.config.ts`。||

它还得有 lint、merge、体积检查、人工 review。||

## 组件工程这一层

【操作】

切到：

`/docs/tailwindcss/merge-and-variants`

讲到这里，| 就很自然了。||

你会发现很多成熟 Tailwind 工程，| 最后都会长出三样东西。||

`tailwind-merge`。||

`cva`。||

`tailwind-variants`。||

先讲 `tailwind-merge`。||

这东西绝对不是小补丁。||

只要组件允许外部传 `className`，| 它就是基础设施。||

因为 Tailwind 最终生成 CSS 的顺序，| 并不简单取决于你字符串里谁写在后面。||

所以不做 merge，| 很多覆盖关系就是不稳定的。||

再讲 `cva`。||

它很适合单槽组件。||

按钮。||

徽标。||

输入框。||

这类组件最重要的事情不是“会写 class”。||

而是把变化集中。||

tone。||

size。||

state。||

defaultVariants。||

compoundVariants。||

这些一旦集中，| 组件立刻就清楚很多。||

`tailwind-variants` 更适合谁。||

更适合多槽组件。||

卡片。||

菜单。||

弹窗。||

Popover。||

当组件有 header、body、footer、icon、trigger 这些槽位时，| slots 的表达就很有价值。||

所以这里不是站队。||

而是问题匹配。||

单槽，| `cva`。||

多槽，| `tv`。||

这就够了。||

这些工具共同做的一件事，| 就是把原子类从“页面写法”抬升到“组件 API”。||

组件工程差不多就是从这里开始真正立住的。||

【操作】

切到：

`/docs/tailwindcss/shadcn-ui`

shadcn/ui 为什么会有这么大影响。||

不是因为它视觉有多好看。||

而是因为它把一套模式推成了事实标准。||

无样式交互 primitives。||

Tailwind 做视觉。||

`tailwind-merge` 做覆盖。||

`cva` 或 variants 做变体。||

最后不是装一个黑盒包。||

而是把源码直接放进项目里。||

这个变化很重要。||

因为它意味着：| 组件所有权回到了团队自己手里。||

以前很多团队对组件库的理解是“装一个包”。||

现在越来越多团队会觉得，| 组件库更像是“把一套成熟模式放进仓库，然后自己维护”。||

所以今天 Headless 这条路才会越来越合理。||

Radix 不是唯一选项。||

Headless UI、Ark UI、Ariakit、React Aria 都可以。||

关键不是选谁。||

关键是你有没有把交互层和视觉层拆开。||

## 样式隔离和边界

【操作】

切到：

`/docs/tailwindcss/style-isolation`

这一页很现实。||

普通应用页面里，| Tailwind 往往问题不大。||

但只要你进入组件库、微前端、Widget、第三方嵌入，| 样式边界马上就会变成核心问题。||

这里没有银弹。||

最轻量的是命名空间。||

也就是 `prefix` 加 `important`。||

再往上是编译期哈希。||

比如 CSS Modules、vanilla-extract。||

再强一点是作用域容器。||

比如 `data-*`、`:where`、未来的 `@scope`。||

最强的是 Shadow DOM 或 iframe。||

还有一个很多人会漏掉的点。||

就是 preflight。||

很多冲突根本不是 utility 本身造成的。||

是 reset 先污染了宿主。||

所以样式隔离不是“选一个更高级的技术”。||

说到底，| 就是根据边界强度，| 选合适的成本。||

## Tailwind 和 UnoCSS 怎么选

【操作】

切到：

`/docs/tailwindcss/tailwind-vs-unocss`

这页别讲成站队。||

它不是先进性比较。||

它是维护模式比较。||

Tailwind 的强项是什么。||

成熟生态。||

官方插件。||

IDE 友好。||

`tailwind-merge` 完整。||

社区范式成熟。||

所以如果一个团队不想自养规则系统、提示系统、merge 逻辑，| Tailwind 会更稳。||

UnoCSS 的强项是什么。||

规则自由度。||

preset 机制。||

DSL 能力。||

即时模式很快。||

所以如果一个团队就是想把样式语言做成内部规则库，| 而且愿意长期维护，| UnoCSS 会很强。||

所以问题根本不是谁更好。||

问题是：| 你到底想复用现成资产，| 还是想自己养一套规则语言。||

## AI 为什么会放大 Tailwind 的价值

【操作】

切到：

`/docs/tailwindcss/ai-friendly-and-demos`

这一页就是今天主题的核心。||

原子类为什么这么适合 AI。||

因为它稳定。||

可组合。||

低歧义。||

词汇空间相对有限。||

模型很擅长在这种空间里做组合。||

但一定要记住一句。||

>>> 适合生成，| 不等于适合交付。||

如果没有护栏，| AI 只会更快地写出一堆“看起来差不多”的代码。||

乱写裸色值。||

乱拼动态类。||

乱用 `group`、`peer`。||

最后把问题留给后面的人收拾。||

所以正确做法不是“让 AI 自由写 Tailwind”。||

而是给它完整上下文。||

告诉它有哪些 tokens。||

有哪些已有组件。||

哪些状态必须走 `aria-*` 或 `data-*`。||

哪些写法禁止出现。||

然后输出完以后，| 还得经过 `cn`、`tailwind-merge`、lint、build 和人工抽查。||

当这些护栏都在的时候，| AI 才会真的变成生产力。||

这里还有一个很值得讲的反转。||

很多人以为 AI 出来以后，| 工程约束没那么重要了。||

实际上恰恰相反。||

AI 出现以后，| 工程约束只会更重要。||

因为原来人类写乱，| 速度还有限。||

现在 AI 写乱，| 会更快，| 规模更大。||

所以在今天，| 原子化 CSS 的价值比几年前更强。||

不是因为类名更流行。||

而是因为它更适合作为一门可约束、可验证、可生成的样式语言。||

## `weapp-tailwindcss` 为什么是关键落地点

讲到这里，| 我想把 `weapp-tailwindcss` 单独拎出来说一下。||

因为很多人第一次看到它，| 会下意识以为它只是一个兼容插件。||

好像它做的事就是：| 把 Tailwind 搞到小程序里。||

其实完全不是这么回事。||

它在做的是一条工程通路。||

它不是只服务一个 demo。||

也不是只绑定一个框架。||

它覆盖的是一整个问题域。||

从仓库里你能看到很多构建工具基底。||

`webpack`、`vite`、`rspack`、`rollup`、`rolldown`、`gulp`。||

也能看到很多开发路线。||

`uni-app`、`taro`、原生小程序、`weapp-vite`、`uni-app x`。||

这说明它不是在告诉你：| 只能在某个模板里抄一遍。||

它是在告诉你：| 这条链路是可以复用、可以迁移、可以工程化沉淀的。||

再往下看，| 它也不是“有个 README 就完了”。||

它有 demo。||

有模板。||

有文档。||

有 Skill。||

还有 watch、HMR、benchmark 这些很工程的东西。||

所以我今天很想把这件事讲透。||

AI 能接进来，| 不是因为 AI 神奇。||

而是因为这里已经有了一条足够稳的工程通路。||

AI 只是把最上游那段重复劳动提速了。||

## 最后的结论

【操作】

结束前切到：

`/docs/tailwindcss/demos`

最后我收一下。||

如果今天这场直播，| 你最后只带走三句话，| 我希望是这三句。||

第一，| 样式方案没有唯一正确答案，| 只有场景匹配。||

第二，| Tailwind 更大的价值，| 不是少写 CSS。||

它更大的价值是：| 更容易把 tokens、variants、merge、组件语义和构建验证串成一条完整链路。||

第三，| 原子化 CSS 只有在建立约束以后，| 才会从“快写法”升级成“稳定工作流”。||

如果再把今天这场直播压成一句最核心的话。||

别再把小程序样式开发当成手写 CSS 的体力活了。||

把它变成一句可以描述的意图。||

让 AI 用 Tailwind 把它说出来。||

再让 `weapp-tailwindcss` 帮你翻译到小程序里。||

以前我们写的是像素。||

今天我们更应该写意图。||

以前我们从属性开始。||

今天我们可以从界面开始。||

以前从想到到看到，| 中间隔着很多轮试错。||

今天这条链路，| 有机会被压缩成一句话、几组类名、一次预览。||

如果你今天回去想自己试，| 我建议就走三步。||

第一，| 先跑一个仓库里的 demo。||

第二，| 找一个真实业务里很常见的小模块。||

第三，| 用一句真实需求试一次，| 让 AI 出 Tailwind 类名，| 再交给 `weapp-tailwindcss` 去落地。||

你只要亲手把这条链路跑通一次，| 你就会知道，| 这不是概念。||

它是真的能把开发效率往前推一截的新范式。||

如果大家愿意，| 后面我也可以继续做第二场。||

把团队规范、Skill、动态 class、运行时合并、`uni-app x` 跨端这些更细的话题再往下讲。||

今天这场直播就到这里。||

谢谢大家。||

## 备用补充

下面这些不是主线必讲。||

但如果你直播的时候想多讲一点，| 或者答疑想往深里走，| 可以直接接。||

### `/docs/tailwindcss/postcss`

如果直播间里偏工程的朋友比较多，| 这页可以补。||

因为很多人知道 Tailwind，| 但不一定意识到它为什么能自然长进现代 CSS 工具链。||

PostCSS 就是关键基础设施之一。||

它把 CSS 处理变成了一条 AST 管线。||

自动补前缀。||

语法降级。||

压缩。||

插件扩展。||

都能稳定插进去。||

所以现代样式工程的进步，| 不只是浏览器原生能力在进步。||

工具链也在不断补工程能力。||

### `/docs/tailwindcss/bem-and-oocss`

这页适合讲给传统前端背景比较重的朋友。||

目的不是让大家回去写 BEM。||

而是让大家意识到：| 前端很早就在认真处理样式混乱问题了。||

BEM 和 OOCSS 的价值，| 是把命名和结构当成工程手段。||

今天 Tailwind 不是反工程。||

它只是把过去靠命名纪律承载的一部分信息，| 转移到了 token、builder、variants 和工具链里。||
