## 痛点

这也就是，为什么这个题值得专门讲一场。||

因为很多人平时嘴上说自己在做前端，| 实际上大把时间花在样式搬运上。||

尤其是小程序。||

你改一个间距，| 切一次设计稿。|
你改一个阴影，| 切一次 IDE。|
你再看结果，| 还得切一次开发者工具或者真机。|
最后一天时间过去了，| 业务逻辑没推进多少，| 样式倒是调了半天。||

这件事为什么一直慢。||

我觉得核心就三个字。| >>> 不划算。||

投入和产出不划算。||

具体一点说，| 是三个问题。||

第一，| >>> 表达成本高。||

设计稿给你的是一个视觉结果。|
但你落代码的时候，| 要自己把它拆成颜色、字号、间距、圆角、边框、阴影、布局、状态。|
这个过程，| 本质上就是把视觉结果翻译成很多零碎属性。||

第二，| >>> 试错成本高。||

很多时候你知道页面不对，| 但你不一定马上知道到底是哪不对。|
是太重了，| 太硬了，| 太平了，| 还是层次不够。|
于是你就只能继续调。||

第三，| >>> 反馈链路长。||

尤其在小程序这里，| 还叠加了构建工具、运行端、框架差异、单位适配、动态 class、生效顺序这些问题。|
所以它不是简单的“写样式”，| 而是一整条很碎的反馈链路。||

这就是为什么我一直觉得，| 问题不是大家不会写。|
问题是，| 我们还在用一种特别低效的方式写。||

如果样式开发一直围绕手写底层属性，| 那 AI 很难稳定帮上忙。|
但如果样式本身是一种可枚举、可约束、可组合的语言，| AI 就会非常顺手。||

Tailwind，| 恰好就是这个中间层。||

## 样式方案演进

【操作】

切到：

`/docs/tailwindcss/history`

接下来我先花一点时间，| 把前端样式方案的演进链路捋一遍。||

因为如果不看历史，| 很多人会误以为 Tailwind 只是一次审美偏好切换。|
但其实不是。|
它接住的是前面很多年累积下来的工程问题。||

先看最早的阶段。||

【操作】

切到：

`/docs/tailwindcss/history/raw-css`

最早当然是 Raw CSS。||

优点非常明显。| 直接，| 简单，| 没门槛。||

问题也非常明显。| 全局污染，| 覆盖链条不透明，| 命名冲突，| 项目一大就很难控制。||

后来有了 BEM、OOCSS 这些方法论。||

它们不是在发明新技术。|
它们是在试图解决一个现实问题。|
如果大家都写全局 CSS，| 那能不能至少靠命名和结构约定，| 让混乱晚一点到来。||

再往后是预处理器。||

【操作】

切到：

`/docs/tailwindcss/history/preprocessors`

Sass、Less 把变量、混入、函数、嵌套这些能力带进来。|
样式第一次更像工程代码。||

但问题并没有消失。|
只是换了个样子。|
因为它的本质还是全局 CSS。|
你只是用更强的语言，| 去生成同样可能失控的样式。||

再往后是 CSS Modules。||

【操作】

切到：

`/docs/tailwindcss/history/css-modules`

这个阶段最关键的变化，| 是作用域开始由编译器保障，| 而不是完全靠人脑维护命名。||

哈希类名一上来，| 全局污染立刻减轻很多。|
所以 CSS Modules 对中大型应用和组件库，| 意义非常大。||

然后是 CSS-in-JS。||

【操作】

切到：

`/docs/tailwindcss/history/css-in-js`

这个阶段的重点，| 是把样式和组件状态绑得更紧。||

这对于多主题、多品牌、动态样式很多的设计系统，| 非常有吸引力。|
但代价也很真实。|
运行时成本。|
SSR 注水。|
构建链复杂度。|
调试体验。|
这些都会上来。||

最后才是 Utility-first。||

【操作】

切到：

`/docs/tailwindcss/history/utility-first`

也就是 Tailwind、UnoCSS 这一类。||

我要强调一下。| 它们不是凭空出现的。|
它们其实是在接前面这些阶段留下来的共同问题。||

大家慢慢发现，| 真正难的不是“样式能不能写出来”。|
真正难的是“样式约束能不能被稳定复用和审阅”。||

所以组织方式开始变化。||

从命名 class，| 转向组合 class。|
从局部抽象，| 转向 tokens 和 variants。|
从运行时解释，| 转向构建期裁剪和生成。||

但这里特别容易讲错。||

我要直接说清楚：|
这不是一条线性替代史。||

不是说 Tailwind 出现了，| 前面的方案就都输了。|
现实更接近多方案并行。||

所以我们做技术选型，| 不应该问“谁赢了”。|
我们应该问的是，| >>> 什么场景下，谁更合适。||

如果你想顺手补一页组件工程的延伸，| 这里可以再带一句 Headless 和 tokens。||

【操作】

切到：

`/docs/tailwindcss/history/headless-tokens`

这一页你可以简单带一下。|
就是今天很多成熟团队，| 已经不再把样式问题看成“class 写法问题”。|
而是把它放回到 tokens、无样式交互 primitives、variants、builder 这些工程层来处理。||

## Tailwind 到底是什么

【操作】

切到：

`/docs/tailwindcss/tailwind-core`

讲完背景，| 我们再看 Tailwind 本身。||

很多人第一次接触 Tailwind，| 会把它理解成“把 CSS 写进 className 里”。||

这个理解对入门有帮助。|
但对工程理解，| 其实远远不够。||

因为 Tailwind 真正的核心，| 不是 class 这个表面。|
而是它背后的整条链路。||

我把它拆成三段。||

第一段，| 入口。||

入口不是 class。| 入口是 tokens。||

颜色、间距、圆角、字号、阴影、动效，| 这些值应该先被收敛成 token。|
而不是等到业务组件里，| 再临时决定。||

第二段，| 中段。||

中段是生成机制。||

Tailwind 通过 content 扫描模板里的类名。|
JIT 只生成真正需要的 CSS。|
插件可以扩展规则。|
`@layer` 可以管理层级。|
`@apply` 可以做稳定复用。||

第三段，| 出口。||

出口不是 utility 本身。| 出口是组件语义。||

也就是说，| 最后落到业务里的，| 不应该是一坨越来越长的 className。|
而应该是有默认值、有变体、有边界的组件 API。||

这就是为什么我会说，| Tailwind 真正解决的，| 不只是“少写 CSS”。||

它真正解决的是：||

怎么让设计约束、类名组合、组件语义和构建产物之间，| 更容易对齐。||

所以 Tailwind 真正厉害的地方，| 不是它让你少打几个字。|
而是它把样式从一种太自由的描述，| 压成了一种更容易约束、复用、生成、审阅的语言。||

## 为什么很多团队会用乱

【操作】

切到：

`/docs/tailwindcss/best-practices`

但问题来了。||

既然它这么强，| 为什么很多团队用了 Tailwind 之后，| 还是会越来越乱。||

原因很简单。| 它太容易开始了。||

你装完几乎马上就能写页面。|
反馈又快。|
于是很多团队会直接跳过中间建设。||

也就是：||

tokens 没抽。|
builder 没建。|
merge 没接。|
content 没收紧。|
评审规则也没立。||

这种情况下，| Tailwind 前期当然很爽。|
但后面问题会慢慢全部回来。||

类名越来越长。|
关系类越来越深。|
状态散落在很多业务组件里。|
动态类开始用字符串拼接。|
评审时每段代码都像是“能跑”，| 但很难判断是不是长期正确。||

所以 Tailwind 最大的误区，| 就是让团队误以为：|
它这么快，| 所以不需要结构。||

实际上，| 它特别需要结构。|
而且这层结构，| 比传统 CSS 时代更重要。||

我觉得至少要有四件事。||

第一，| tokens。||

不要在业务组件里到处写裸色值、裸字号、裸间距。|
应该先有一份设计 token。|
这样未来改主题、改品牌、做暗色模式，| 改的是映射，| 不是全项目搜字符串。||

第二，| variants。||

只要组件有 `size`、`tone`、`state`、`intent` 这些维度，| 就不要把它们散落在业务组件里。|
应该集中到 builder 层。|
比如 `cva` 或 `tailwind-variants`。||

第三，| merge。||

这个点特别关键。||

`tailwind-merge` 不是一个小补丁。|
它解决的是 Tailwind 类冲突和外部覆盖的可预测性问题。||

因为 Tailwind 最终生成 CSS 的顺序，| 并不简单取决于你 class 字符串的顺序。|
如果没有 merge，| 组件允许外部传 `className` 的那一刻起，| 就已经埋下了不稳定因素。||

第四，| content 和构建验证。||

Tailwind 的问题很多时候不在运行时，| 而在构建时。|
扫描范围太宽，| 动态类自由拼接，| 关系类过深，| 都会让产物失控。||

所以一套成熟的 Tailwind 工程，| 不只是有 `tailwind.config.ts`。|
还必须有 lint、merge、体积检查、人工 review 这些验证环节。||

## 组件工程这一层

【操作】

切到：

`/docs/tailwindcss/merge-and-variants`

讲到这里，| 我们就能顺着看 `tailwind-merge`、`cva` 和 `tailwind-variants` 这一层。||

我觉得它们就是 Tailwind 真正进入组件工程的桥。||

`tailwind-merge` 负责冲突去重和覆盖关系。|
`cva` 更适合单槽组件。|
比如按钮、徽标、输入框。|
`tailwind-variants` 更适合多槽组件。|
比如卡片、菜单、弹窗。||

它们共同做的事情，| 是把原子类从“页面写法”，| 抬升到“组件 API”。||

也正因为这样，| shadcn/ui 才会有那么大影响。||

【操作】

切到：

`/docs/tailwindcss/shadcn-ui`

它最重要的地方，| 不是它的视觉风格。|
而是它推广了一种模式。||

底层用 Radix 这类无样式交互 primitives。|
视觉层用 Tailwind。|
变体层用 `cva` 或 `tailwind-variants`。|
覆盖关系用 `tailwind-merge`。|
最后把源码直接复制进项目，| 而不是单纯依赖一个 npm 包。||

这个模式最重要的变化，| 就是组件所有权回到了团队自己手里。||

以前很多团队会觉得组件库就是“装一个包”。|
现在越来越多团队会觉得，| 组件库应该是“把一套成熟模式落到仓库里，然后自己维护”。||

关键不是你是不是非得用 shadcn/ui。|
关键是你有没有真正把交互层和视觉层拆开。||

## 样式隔离和边界

【操作】

切到：

`/docs/tailwindcss/style-isolation`

接下来讲一个经常被忽略，| 但其实非常现实的话题。||

样式隔离。||

Tailwind 用在普通应用页面里，| 问题往往不大。|
但一旦进入组件库、微前端、Widget、第三方嵌入，| 样式边界立刻就会变成核心问题。||

这里没有银弹。||

最轻量的是命名空间。| 也就是 `prefix` 加 `important`。|
再往上是编译期哈希。| 比如 CSS Modules、vanilla-extract。|
再强一点是作用域容器。| 比如 `data-*`、`:where`，以及未来的 `@scope`。|
最强的是 Shadow DOM 或 iframe。||

还有一条经常被忽略的，| 是 preflight 控制。|
很多冲突其实不是 utility 本身造成的，| 而是 reset 先把宿主页面改了。||

所以样式隔离不是“选一个更高级的技术”。|
它是“根据边界强度选合适的成本”。||

## Tailwind 和 UnoCSS 怎么选

【操作】

切到：

`/docs/tailwindcss/tailwind-vs-unocss`

再讲一个今天大家一定会问的问题。||

Tailwind 和 UnoCSS 怎么选。||

我觉得从工程视角看，| 它们的差异其实很清楚。||

Tailwind 的优势，| 是现成生态、IDE 支持、官方插件、merge 规则和大量社区范式。||

UnoCSS 的优势，| 是规则自由度、preset 机制和定制能力。||

所以如果一个团队不想自养规则系统、提示系统和 merge 逻辑，| Tailwind 会更稳。||

如果一个团队本身就想把样式语言做成内部 DSL，| 而且愿意长期维护 preset 仓库，| 那 UnoCSS 会非常强。||

这不是技术先进性比较。|
这本质上是维护模式比较。||

## AI 为什么会放大 Tailwind 的价值

【操作】

切到：

`/docs/tailwindcss/ai-friendly-and-demos`

最后讲 AI。||

我觉得 AI 会让 Tailwind 这条路线的工程价值，| 变得更明显。||

原因很简单。||

原子类天然适合模型生成。|
它们结构稳定。|
组合清晰。|
词汇一致。|
不像大量自定义样式那样高度随意。||

但一定要记住。||

适合生成，| 不等于适合交付。||

如果没有约束，| AI 只会更快地产出一堆“看起来差不多”的代码。|
它会乱写裸色值。|
乱拼动态类。|
乱用 `group` 和 `peer`。|
然后把问题丢给后面的人收拾。||

所以正确做法，| 不是“让 AI 自由写 Tailwind”。|
而是给它完整上下文。||

告诉它有哪些 tokens。|
有哪些已有组件。|
哪些状态必须走 `aria-*` 或 `data-*`。|
哪些写法禁止出现。||

输出后还必须经过 `cn`、`tailwind-merge`、lint、build 和人工抽查。||

当这些护栏都在的时候，| AI 才会真正变成生产力。||

而且这里有一个很有意思的反转。||

很多人以为 AI 会让工程约束没那么重要。|
实际上恰恰相反。|
AI 出现以后，| 工程约束只会更重要。||

因为原来人类写乱，| 速度还有限。|
现在 AI 写乱，| 会更快，| 更大规模。||

所以原子化 CSS 在今天的意义，| 比几年前更强。|
不是因为类名更流行了。|
而是因为它更适合作为一门 >>> 可约束、可验证、可生成的样式语言。||

## `weapp-tailwindcss` 为什么是关键落地点

讲到这里，| 我想把 `weapp-tailwindcss` 的位置单独拎出来说一下。||

因为很多人第一次看到它，| 会下意识把它理解成一个兼容插件。|
好像它做的事情就是，| 把 Tailwind 搞到小程序里去。||

但如果你认真看这个仓库，| 你会发现完全不是这个量级。||

它在做的是一条工程通路。||

为什么这么说。||

因为它不是只服务一个 demo。|
也不是只绑定一个框架。|
它覆盖的是一整个问题域。||

从仓库物料能看到，| 它支持多种构建工具基底，| 也支持多种开发路线。|
你会看到 `webpack`、`vite`、`rspack`、`rollup`、`rolldown`、`gulp`。|
你也会看到 `uni-app`、`taro`、原生小程序、`weapp-vite`、`uni-app x`。||

这说明它不是告诉你，| 只能在某个模板里抄一遍。|
它是在告诉你，| 这条链路是可以被复用、被迁移、被工程化沉淀的。||

再往下看，| 这个仓库也不是“有个 README 就算完”。|
它有 demo，| 有模板，| 有文档，| 有 Skill，| 还有 watch、HMR、benchmark 这些很工程的东西。||

这就说明，| 作者关注的不是“能不能过编译”。|
