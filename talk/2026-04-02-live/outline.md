# 直播大纲

## 直播标题

小程序还能这么写？AI 出样式，Tailwind 跑全端

## 直播简介

2026 年了，还在小程序里一行一行写 `margin-left: 20rpx`？该醒醒了。这场直播带你把 `AI + Tailwind CSS + weapp-tailwindcss` 串成一条真正可用的开发链路：一句话描述界面，AI 先出原子类名，`weapp-tailwindcss` 再自动转译成小程序可运行样式，最后直接在真机和开发者工具里看到结果。全程围绕真实仓库展开，现场演示 AI 出样式、插件转译、多框架能力、Skill 工作流和进阶排障，尽量做到“不手写一行 CSS”。

## 直播目标

- 让观众理解为什么 Tailwind 特别适合和 AI 配合
- 让观众知道 `weapp-tailwindcss` 不只是“兼容插件”，而是完整工程方案
- 让观众看到至少一条从零到页面的最短落地路径
- 让观众带走可复制的提示词、配置心智和避坑顺序

## 每一段讲什么

### 1. 开场钩子

- 问观众三个问题：
  - 你上周写了多少行样式？
  - 你有多少时间花在改 `padding/margin/radius/shadow`？
  - 你有没有把“UI 还原”当成体力活？
- 抛出观点：
  - 现在真正该手写的不是 CSS，而是需求约束与组件意图

### 2. 问题重述

- 小程序样式开发慢，不只是因为 CSS 多
- 真正慢在三件事：
  - 表达成本高
  - 试错成本高
  - 跨端一致性差
- 过渡：
  - 如果样式能被表达成“结构化语言”，AI 就有机会接管一大半重复劳动

### 3. 新范式演示

- 现场对 AI 说：
  - “做一个渐变卡片，圆角，柔和阴影，标题 32rpx，副标题 24rpx，底部一个高亮 CTA”
- 展示 AI 直接返回 Tailwind 类名
- 展示这些类名放进页面后，由 `weapp-tailwindcss` 自动转译并在小程序里生效

### 4. 为什么 AI + Tailwind 天作之合

- Tailwind 类名天然是“可枚举、可约束、可组合”的
- AI 最怕的是开放式视觉描述，最擅长的是半结构化输出
- “渐变卡片带阴影”这种描述，AI 生成一段原子类，比让它生成一整份手写 CSS 更稳定
- 这时 `weapp-tailwindcss` 的意义不是“写样式”，而是“把这套语言翻译到小程序”

### 5. 项目全景

- 仓库亮点建议讲 6 个：
  - 支持 `webpack`、`vite`、`rspack`、`rollup`、`rolldown`、`gulp`
  - 仓库自带多框架 demo 与模板
  - Skill 已产品化
  - 有 watch/HMR 回归
  - 有 benchmark
  - `uni-app x` 已有 5 端故事

### 6. 实操 Round 1

- 从 `demo/uni-app-tailwindcss-v4` 开始
- 重点展示：
  - `postinstall: "weapp-tw patch"`
  - `vite.config.ts` 注册插件
  - `src/main.css` 中的 Tailwind 入口
  - 页面里直接上类名
- 目标：
  - 让观众相信“这条链路并不复杂”

### 7. 实操 Round 2

- 从 Skill 文档出发
- 展示一个提示词如何被拆成：
  - 框架
  - 构建器
  - 目标端
  - Tailwind 版本
  - 包管理器
- 再展示 AI 不只是写页面，还能：
  - 生成配置
  - 提醒 `patch`
  - 给验证步骤
  - 给回滚方案

### 8. 进阶技巧

- 任意值与 `rpx`：
  - `text-[22rpx]` 这类写法什么时候会歧义
- 动态 class：
  - 不要自由拼接半截 token
  - 要枚举
- `space-y` / `space-x`：
  - 小程序标签限制
- 运行时合并：
  - `twMerge`、`cva`、`cn`
- `uni-app x`：
  - 为什么这让“Tailwind 跑全端”这句话成立

### 9. 工程视角收尾

- 引用仓库内 2026-02-13 的 HMR 报告和 2026-02-23 的框架对比报告
- 核心结论不是“某框架必赢”
- 核心结论是：
  - 作者已经把“开发体验”和“多方案适配”当成工程问题在做
  - 这意味着观众今天学到的不只是一个插件，而是一条新工作流
