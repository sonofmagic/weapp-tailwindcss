## 05:00-08:00 问题重述

### 要说的话

我觉得小程序样式开发慢，核心不只是因为 CSS 多。

真正慢在三件事。

第一，表达成本高。  
设计稿给的是视觉结果，但开发实现时，要自己拆成颜色、阴影、布局、间距、字号、状态，这一步很机械。

第二，试错成本高。  
你经常知道它“不对”，但不确定到底是圆角不对、层次不对，还是按钮不够突出。于是你要不断微调。

第三，跨端一致性差。  
你今天可能是 `uni-app`，明天可能是 `taro`，后天可能还要面对原生小程序或者 `uni-app x`。  
每条链路都不完全一样，开发者就会被迫回到很底层的兼容细节里。

所以问题不是“大家不努力”，而是传统样式写法本身就不够适合现在的开发节奏。

如果样式还是以手写属性为中心，那 AI 很难稳定介入。  
但如果样式变成一种带约束的语言，那 AI 就能变得很好用。  
Tailwind 恰好就是这样一种语言。

- 左边：`手写样式属性`
- 右边：`描述界面意图 -> AI 生成原子类`

你不是先想 `border-radius` 写多少、`box-shadow` 写几层，而是先想“我要一个什么感觉的卡片”，然后让 AI 用 Tailwind 把它说出来。

## 15:00-23:00 为什么 AI + Tailwind 天作之合

### 要说的话

讲到这里，很多人会问，为什么偏偏是 Tailwind？

原因其实很朴素。  
Tailwind 的类名，本质上是一种半结构化语言。  
它不像传统 CSS 那么自由散漫，它有固定词汇、固定模式、固定组合关系。

比如：

- `rounded-2xl`
- `shadow-lg`
- `px-4`
- `py-3`
- `text-[28rpx]`
- `from-cyan-500`
- `to-blue-600`

这些东西都很像“有限语法里的词块”。  
而 AI 最擅长的，就是在有限语法空间里做组合。

所以你会发现，让 AI 写传统 CSS，经常显得过度设计。  
但让 AI 写 Tailwind 类名，它反而很稳定，因为它是在拼一套受约束的语言。

换句话说，Tailwind 不是为了少写字。  
Tailwind 的真正价值，是把样式从“自由属性集合”变成“可被机器理解的语言”。

这也是为什么今天这个主题成立。  
AI 并不是直接对接小程序，而是先对接 Tailwind 这层表达语言。  
`weapp-tailwindcss` 再去负责翻译，让这些类名在小程序环境里真正可运行。

所以整个链路可以理解成三层：

1. 你负责描述意图
2. AI 负责生成 Tailwind 类名
3. `weapp-tailwindcss` 负责翻译到小程序

这三层各干各的，整个系统就稳定了。

## 56:00-64:00 进阶技巧

### 要说的话

接下来我补几个很容易踩坑，但特别值得在直播里讲清楚的点。

第一个，任意值和 `rpx`。

像 `text-[22rpx]` 这种写法，在小程序里很实用。  
但是某些场景会有二义性，这时候要更明确一点，比如写成 `text-[length:22rpx]`。  
所以任意值不是不能用，而是要知道什么时候写得更明确。

第二个，动态 class。

这一点我特别建议大家记住。  
不要写 `bg-${color}-500` 这种半截拼接。  
不管是人写，还是 AI 写，这种方式都不稳。  
更好的方式是枚举，让它从完整字面量里选。

第三个，`space-y` 和 `space-x`。

这类类名在 Web 里大家很顺手，但在小程序里它依赖子组合器，不是天然对所有标签都一样生效。  
所以仓库里给出的建议顺序是：

1. 先改结构
2. 再看 `virtualHost`
3. 最后才扩配置

第四个，运行时合并。

如果你是做组件库，或者页面里存在外部类名覆盖，那 `twMerge`、`cva`、`cn` 这些东西会非常重要。  
它不是为了炫技，而是为了让“默认类 + 覆盖类 + 冲突去重”这件事更稳。

第五个，`uni-app x`。

README 已经明确写了，`weapp-tailwindcss@4.2.x` 支持 `uni-app x` 同时构建 `Web`、小程序、`Android`、`iOS`、鸿蒙。  
这意味着今天我们讲的不是“小程序里能不能跑 Tailwind”这么简单，实际上是在讲一个更大的方向：  
能不能把 Tailwind 作为多端统一样式语言来使用。

## 备用话术

### 观众问：AI 生成的类名是不是很乱

可以这么说：

会，所以不能无脑相信。  
但我们今天的目标不是让 AI 一步到位，而是让它先给出一个高质量起点。  
然后你再通过枚举、`twMerge`、组件抽象去收敛。

### 观众问：这是不是只适合 Demo

可以这么说：

如果它只是 Demo，我不会拿来讲。  
这个仓库里同时有模板、Skill、E2E、watch/HMR 报告、benchmark，这说明它已经在解决工程问题，不只是展示效果。

### 观众问：为什么不用 AI 直接写 CSS

可以这么说：

能写，但稳定性和可维护性通常不如 Tailwind 类名。  
Tailwind 更像一种受约束的语言，而 AI 更擅长在受约束的语言里输出高质量结果。

### 观众问：是不是所有项目都适合全量 Tailwind

可以这么说：

不一定。  
我更建议从收益高的页面开始试，比如卡片、列表、营销页、活动页。  
先把高频重复样式拿下，再考虑是否扩大范围。
